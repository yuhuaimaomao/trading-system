# 第一部分：系统全景

## 1. 系统全景图

### 1.1 一句话概括

这是一个 **A 股量化交易辅助系统**。它运行在 macOS 上，通过网络连接到 Windows 电脑上的 QMT（券商量化终端），每天自动完成以下工作：

- **盘后**：采集全市场数据 → AI 多维度分析写复盘报告 → 策略管线筛选候选股 → AI 研判生成买入信号
- **盘前**：汇总隔夜资讯 + 宏观变化 → AI 生成盘前简报 + 板块倾向分析 → 校准昨日信号
- **盘中**：双引擎盯盘扫描 → 情景贝叶斯预判 → 判断大盘/板块/个股 → 模拟盘自动执行 → 风控止损止盈
- **事后**：复盘预测核验 → 双域审计（策略+盯盘）→ 发现决策偏差 → 生成改进建议

所有消息通过 Telegram 推送到你手机上，你也可以通过 Telegram 回复来控制实盘操作。

### 1.4 AI 模型配置

系统支持**每个业务独立配置AI模型**，通过环境变量控制：

| 业务 | 环境变量 | 用途 |
|------|---------|------|
| 默认 | `AI_MODEL` | 未配置独立模型的业务使用 |
| 复盘报告 | `AI_MODEL_REVIEW` | 盘后复盘分析 |
| 趋势筛选 | `AI_MODEL_SCREENING` | 策略选股AI研判 |
| 早盘简报 | `AI_MODEL_MORNING` | 盘前简报+板块倾向 |
| 盯盘通用 | `AI_MODEL_WATCHER` | 盯盘通用AI |
| 追高二判 | `AI_MODEL_WATCHER_CHASE` | 追高/被拒的二次判断 |
| 换仓评估 | `AI_MODEL_WATCHER_SWAP` | 换仓评估 |
| 指数波动 | `AI_MODEL_WATCHER_INDEX` | 大盘急拉急跌分析 |
| 被套离场 | `AI_MODEL_WATCHER_TRAPPED` | 深度被套离场分析 |
| 审计 | `AI_MODEL_AUDIT` | 策略+盯盘审计 |

支持 Batch API 模式（`BATCH_ENABLED=true`），用于策略管线批量AI研判。支持多Provider（DashScope/DeepSeek），自动检测模型前缀选择API端点。

## 2. 一天的时间线

系统按照交易日的自然节律运行：

```
08:00 ─ 电报采集开始（每20分钟，持续到20:00）
08:30 ─ 盘前简报：读取复盘报告+隔夜宏观+电报+早报
         → AI校准昨日pending信号（取消/调整/降级）
         → AI分析今日板块倾向（聚焦/回避/中性/择优）→写入morning_sector_bias表
         → 推送简报+校准结果到TG
09:00 ─ Telegram监听启动（listen进程，前台阻塞运行）
09:25 ─ 盯盘系统启动（monitor进程，PID文件防多实例）
         → 自动检测QMT Collector是否运行（不在则拉起）
         → 连接Collector TCP socket
         → 恢复昨日持仓+_pos_meta
         → 清理旧快照数据
09:25 ─ 集合竞价结束，第一轮扫描（推送开盘决策汇总）
09:30-11:30 ─ 上午连续竞价，盯盘每分钟扫描
         → 引擎1（趋势跟踪）：信号+复盘推荐+动态板块+回踩机会
         → 引擎2（底部发力）：低涨幅+量能放大+涨幅加速+板块联动（10:00起）
         → 情景预测引擎持续更新8情景概率
11:30-13:00 ─ 午休，盯盘休眠
13:00-14:30 ─ 下午连续竞价（引擎2停止）
14:30 ─ 尾盘决策：逐只判断是否持过夜
15:00 ─ 收盘：保存快照+过期信号+推送持仓报告+清理运行时状态
         → 监听进程退出
18:00 ─ 盘后复盘：采集14个模块→数据统计→AI分析→预测提取→推TG
         → 成功后自动触发策略管线：
           市场宽度→趋势筛选→因子评分→AI研判→信号入库+持仓审查
20:00 ─ 电报采集停止
```

## 3. 数据采集系统

### 3.3 代理与反爬

- **ProxyManager**：管理代理IP池，定时验证可用性
- **ProxyRequester**：封装requests，自动重试、换IP
- **DNS绕过**：`dns_bypass.py`在系统启动时patch Python socket层，绕过Surge/Clash等本地代理工具的DNS劫持（当前临时禁用中，观察curl_cffi连接状态）
- 财联社API（TelegraphCollector）直连，不走代理

### 3.4 三种运行方式

| 方式 | 命令 | 时机 |
|------|------|------|
| 复盘全量采集 | `review`命令内部调用14个采集器 | 每天18:00 |
| 电报定时采集 | `start_news_collection.sh`→`collect --module news` | 每20分钟 |
| 手动指定模块 | `python main.py collect --module market\|events\|macro` | 按需 |

## 4. 盘后复盘管线

### 4.1 它是干嘛的

每天收盘后，自动采集全市场数据→清洗统计→AI写分析报告→提取预测项→推送Telegram。复盘成功后自动触发策略选股管线。

启动：`python main.py review`（cron 每天18:00）。支持 `--analyze-only` 跳过采集直接分析。

### 4.2 完整流程

```
Step 1: 数据采集（14个采集器依次跑）
  ├── 行业板块 + 概念板块
  ├── 个股行情（~5000只，含技术指标入库stock_indicators）
  ├── 强势股 + 龙虎榜 + 涨跌停池
  ├── 大盘指数 + 监管函 + 停复牌 + 股东增减持
  ├── 公告预告 + 隔夜宏观
  └── 早报摘要（CLSDigest）

Step 2: 数据统计（CollectionStatsService）
  ├── 采集完成度检查（是否所有模块成功）
  ├── 板块数据完整性验证
  └── 输出统计报告到日志

Step 3: AI分析（ReviewAnalyzer v3.0 刺客风格）
  ├── 设计理念：程序采集+清洗数据，AI做判断+推演
  ├── 不给AI喂加工过的「得分」，只喂原始数据+环比对比
  ├── 读取数据+环比变化→组装结构化Prompt（~15个格式化函数）
  ├── Function Calling辅助：查股票代码/查板块/查龙虎榜/查历史校准/查经验教训
  └── 调用AI（按AI_MODEL_REVIEW配置）→生成复盘报告→保存到storage/reports/

Step 4: 预测提取（PredictionRepo）
  ├── 从AI报告中解析预测项：指数方向/板块走势/情景判断
  └── 写入review_predictions表（is_verified=false），供次日核验

Step 5: 其他处理
  ├── 涨停股次日表现统计（ZTPerformanceProcessor）
  ├── 板块采集信息更新（SectorInfoProcessor）
  └── 选股跟踪更新（StockTracker）

Step 6: TG推送+自动触发策略管线
  └── 推送到群聊→cmd_strategy()
```

### 4.3 AI分析的数据输入

复盘AI会收到结构化Prompt，包含四大类数据：

**A. 行情背景（~10个格式化模块）**
- 今日大盘概况（涨跌幅、成交量、涨跌家数、振幅）
- 板块轮动（领涨/领跌TOP10、持续性、三日趋势对比）
- 涨停分析（涨停家数、炸板率、连板高度、龙头股、首板质量、炸板未回封统计）
- 龙虎榜（机构/游资动向、买卖席位TOP10）
- 资金流向（主力净流入/流出板块TOP5、北向资金）
- 异动个股（量比暴增、急速拉升、监管关注）

**B. 趋势深度（~3个格式化模块）**
- 强势股追踪（连续涨停、近期新高、多次涨停）
- 三日板块对比（持续性判断）
- 资金集中度（主力资金聚焦方向）

**C. 参考上下文（~4个格式化模块）**
- AI历史校准经验（从strategy_lessons读取的已知问题和规避建议）
- 昨日复盘预测准确率
- 过往推荐追踪表现

**D. 宏观与风险**
- Shibor/国债/汇率/外围市场变化
- 风险标记：炸板池、监管升级、利空公告、财务风险

### 4.4 复盘报告的Function Calling工具

复盘AI在分析过程中可以调用20个Function Calling工具查询实时数据：

| 类别 | 工具 | 用途 |
|------|------|------|
| 基础信息 | search_stock / get_stock_info | 搜索股票代码、查基本面和财务 |
| 板块分析 | search_sector / get_sector_stocks / get_sector_zhongjun | 查板块成分股、找板块龙头（中军） |
| 龙虎榜 | get_lhb_seats | 查席位买卖明细 |
| 风险排查 | get_regulatory_risks | 查监管函/问询函/立案调查 |
| 涨停分析 | get_yesterday_limit_ups / get_hotspot_stocks | 查昨日涨停股/今日热点 |
| 资金流向 | get_market_cap / get_unusual_stocks | 查市值/异动股发现 |
| 自我校准 | get_historical_calibration / get_learning_lessons / get_prediction_accuracy | 查历史经验教训和预测准确率 |
| 持仓信号 | get_pending_signals | 查当前pending信号列表 |
| 新闻资讯 | get_cls_digest_news / get_telegraph_news | 查早报/电报 |
| 复盘参考 | get_yesterday_review / get_yesterday_picks_performance | 查昨日复盘/选股表现 |

## 5. 策略选股管线

### 5.1 它是干嘛的

盘后复盘完成后自动运行（也可手动 `python main.py strategy [日期]`）。从全市场~5000只股票中，经过**多层筛选+多因子评分+AI深度研判**，最终产出可交易的买入信号，写入 `trade_signals` 表供次日盯盘使用。

同时AI还会审查现有持仓，给出 hold/reduce/close 建议。

### 5.2 完整筛选漏斗

```
全市场 ~5000 只
    │
    ▼ 步骤0: 市场宽度（MarketBreadth.classify_market_state）
    │   根据涨跌家数判断大盘状态：普涨/分化/普跌/恐慌/连跌修复/超跌末端
    │   恐慌状态→不生成新信号，直接结束
    │
    ▼ 步骤0.5: 加载持仓上下文
    │   从trade_portfolio_snapshots/positions读取模拟盘+实盘持仓
    │   为每只持仓构建HoldingInfo（成本/盈亏/T+1锁定/板块）
    │   构建AccountSummary（总资产/现金/仓位比例）
    │
    ▼ 步骤0.8: 加载复盘上下文（ReviewContext）
    │   从复盘报告提取：市场情绪/主线方向/板块轮动/风险提示
    │
    ▼ 步骤1: 趋势筛选（TrendScreener）
    │   基础查询→排除严重风险→硬关卡→板块关卡→板块黑名单→
    │   运行40+因子→场景匹配→评分→排名→模式判定
    │   通过→候选池（通常50-200只）
    │
    ▼ 步骤1.5: 加载昨日遗留（Legacy）
    │   昨天AI推荐但未入选的候选，今天重新评估
    │
    ▼ 步骤2: 画像富化（ProfileBuilder）
    │   ⭐StockScore→StockProfile：
    │   补历史数据(60天)+RPS计算+板块参照+估值+快讯+技术指标+风险标记
    │
    ▼ 步骤3: AI研判（AIAdvisor）
    │   每个候选→构建Prompt→调用AI（支持Batch API批量模式）→解析JSON响应
    │   AI输出: 看到什么/担心什么/决定因素/判定/买入区间/止损止盈/定价逻辑
    │
    ▼ 入库
        strategy_funnel: 每层通过情况
        strategy_ai_decisions: AI研判详情（含reasoning和pricing子对象）
        trade_signals: 入选的生成pending信号（含精确买入区间/止损/止盈/评分）
        trade_holdings_review: AI对持仓的审查建议（hold/reduce/close+新止损止盈）
```

### 5.3 趋势筛选详情（TrendScreener）

#### 硬关卡（6道，不通过直接淘汰）
1. **排除ST股**：名称包含ST/退市
2. **排除科创板**：688开头（波动大、门槛高）
3. **排除大跌股**：今日跌幅 > 9.5%
4. **市值门槛**：市值 > 50亿
5. **流动性门槛**：量比 > 0.3
6. **均线关卡**：价格 > MA20 且 MA10 > MA20（中期趋势向上）
7. **涨停标记**：涨停的标记但不排除（供后续场景使用）

#### 板块关卡
- 股票所属的板块中至少有一个近5天在热点榜上
- **板块黑名单**：白酒/银行/保险/券商/房地产/农业/新能源车/环保/消费/食品（这些板块不参与趋势选股）

#### 多因子体系（40+个因子，分为6类）

**量价因子（3个）**：
- 放量启动：量比≥1.5
- 缩量回调：量比≤0.7且-2%<涨跌幅<0%
- 蓄力中：振幅<3%

**资金因子（2个）**：
- 主力介入：主力净额>0且主力比率>3%
- 筹码集中：主力净额>0且小单净额<0

**多日因子（6个）**：
- 强势连阳：近3天全部收阳
- 回踩确认：距5日高点回调<5%
- 趋势延续：近5天MA5>MA10>MA20
- 低波蓄力：ATR14/价格<3%
- 量能放大：5日均量>20日均量×1.2且累计涨幅>0
- 趋势强劲：10日涨幅>5%且夏普比率>1.0

**板块因子（4个）**：
- 板块加持：近5天板块在TOP5内
- 领涨龙头：所属板块内涨幅前3
- 强于板块：涨幅超过板块均值+0.5%
- 资金共振：板块和个股主力净买入均为正

**技术因子**：
- 周线多头：价格>BBI且趋势确认
- 高位运行/低位启动：20日高低区间百分位>80%/<30%
- 量价齐升：涨且量比>1.2
- 板块量价齐升：板块涨>0.5%且资金净流入

#### 场景匹配（11个场景）

| 场景 | 含义 | 加分 |
|------|------|------|
| 低位启动 | 价格在20日低点附近+放量+资金流入 | +8 |
| 突破追涨 | 突破20日高点+量能配合 | +8 |
| 新高突破 | 创60日新高 | +8 |
| 趋势加速 | MA5角度>2+量比>1.5 | +8 |
| 强势反包 | 昨跌今涨且涨幅大于昨日跌幅 | +8 |
| 回踩MA5/MA10/MA20 | 价格回踩关键均线+量缩 | +8 |
| 高位蓄力 | 高位窄幅震荡+主力持续流入 | +8 |
| 强势横盘 | 横盘期间涨幅不跌+量缩 | +8 |
| 底部反弹 | 底部+放量+反转信号 | +8 |
| 均线发散 | MA5/MA10/MA20多头排列+发散 | +8 |
| 趋势行进 | 沿MA5稳步上行 | +8 |

#### 评分体系
```
基础分 = 20
+ 每个通过因子标签 × 5
+ 每个匹配场景 × 8
+ MA5角度调整（角度>0加分最高+10，角度<0扣分最高-10）
+ 位置惩罚（高位放量-8，低位量价齐升+8）
= 最终得分（排序用）
```

#### 模式判定
- **strong**：价格紧贴MA5（偏差≤3%）且MA5角度≥1
- **normal**：价格>MA20且偏离<15%
- 其他：按实际情况标记

## 6. 盘前简报

### 6.1 它是干嘛的

每天早上开盘前，自动汇总隔夜发生的所有重要信息，AI做两件事：
1. **生成盘前简报**：宏观变化+重要电报+风险提醒+持仓关注→推送到TG
2. **校准昨日信号**：根据隔夜变化，对pending信号做 cancel/adjust/downgrade 操作
3. **板块倾向分析**：判断今日该聚焦/回避哪些板块→写入morning_sector_bias表→盯盘系统加载

启动：`python main.py morning`（cron 每天08:30）

### 6.2 简报数据来源

| 板块 | 内容 | 数据来源 |
|------|------|----------|
| 昨日复盘 | 复盘报告全文（含AI研判+板块机会+风险提示） | storage/reports/ |
| 隔夜宏观 | Shibor变化、国债收益率、汇率、外盘涨跌、大宗商品 | macro_daily表 |
| 重要电报 | 昨日收盘→今早的财联社电报（AI重要性≥3的优先） | cls_telegraph表 |
| 早报摘要 | 财联社早报（CLSDigest） | 财联社API |
| 避雷针 | 监管函/问询函/立案调查/减持公告/业绩暴雷 | regulatory_letter/future_announcements/share_holder_change |

### 6.3 AI校准逻辑

AI分析后输出 `<<<ADJUSTMENTS>>>` JSON块，对pending信号执行以下操作：

| 操作 | 效果 | 示例场景 |
|------|------|----------|
| cancel | 信号状态→cancelled | 隔夜重大利空、监管立案 |
| adjust | 修改买入区间/止损价/止盈价/评分 | 隔夜外盘大涨→上调买入区；利空传闻→下调 |
| downgrade | 降低信号评分 | 板块轮动预期变化、外围风险上升 |
| focus | 标记板块为聚焦 | 利好政策、外围大涨→优先买入 |
| avoid | 标记板块为回避 | 利空政策、外围大跌→跳过该板块信号 |
| selective | 标记板块为择优 | 板块分化预期→仅高质量信号可买入 |

校准结果写入 `morning_sector_bias` 表，盯盘系统启动时加载。

### 6.4 板块倾向如何影响盘中交易

盯盘系统加载 `morning_sector_bias` 后，在买入决策中应用：

| bias | 效果 |
|------|------|
| focus | 仓位乘数×1.0~1.3（根据priority），可放宽筛选阈值 |
| avoid | 该板块信号直接跳过，不评估 |
| neutral | 无特殊影响 |
| selective | 仅信号评分特别高（比正常阈值+10分以上）时才买入 |

## 7. 预测核验系统

### 7.1 它是干嘛的

复盘报告中AI会对次日市场做出方向性预测（指数方向、板块走势、市场情景）。`verify-predictions` 命令在次日收盘后，自动对比这些预测 vs 实际市场数据，计算准确率，写入数据库，推送到Telegram。形成复盘→预测→核验→改进的**自我校准闭环**。

启动：`python main.py verify-predictions [--date 日期]`

### 7.2 核验维度

#### 指数预测核验
根据各指数的波动率阈值判断AI预测的方向是否正确：

| 指数 | 偏多阈值 | 单边上涨阈值 | 偏空阈值 | 单边下跌阈值 |
|------|---------|------------|---------|------------|
| 上证指数 | 0~0.3% | >0.8%+阳线 | -0.3%~0 | <-0.8% |
| 深证成指 | 0~0.5% | >1.0% | -0.5%~0 | <-1.0% |
| 创业板指 | 0~1.0% | >1.5% | -1.0%~0 | <-1.5% |
| 科创50 | 0~1.0% | >1.5% | -1.0%~0 | <-1.5%或<-3.0%暴跌 |

同时核验支撑位/阻力位是否被突破。

#### 板块预测核验
在 `sector_industry`/`sector_concept` 表中匹配AI提到的板块，验证方向是否正确：
- **主线延续**：板块继续涨且排名≤10→正确
- **分歧后回流**：板块涨→正确
- **一日游风险**：板块跌或排名>15→正确（AI预判了风险）
- **退潮**：板块跌→正确

#### 情景预测核验
综合指数变化方向+板块准确率（≥60%为通过）。

### 7.3 数据流

```
读: review_predictions表（is_verified=false）
    → 获取下一个交易日数据
    → 匹配实际涨跌幅
    → 逐条判定is_correct
写: review_predictions表（更新is_correct字段）
推: Telegram群聊（准确率汇总+逐条明细）
```

### 7.4 输出格式

```
📊 预测核验报告 2026-06-05（对比 2026-06-06 实际数据）
核验总数：25 条
  指数：8 条 | 板块：12 条 | 情景：5 条
✅ 正确：18 条
❌ 错误：5 条
⚠️ 未匹配：2 条
📈 准确率：78.3%
```

- **主力净额+占比**：>5%大幅流入，>2%温和流入，<-5%大幅流出，<-2%温和流出

#### 板块属性分析（SectorAttrAnalyzer）

- **所属行业**：申万行业分类

# 第四部分：盘中交易

## 9. 盯盘系统概述

盯盘系统（Monitor）是整个系统中**唯一持续运行的实时模块**。交易日09:25-15:00期间每分钟扫描一次，负责：判断市场状态→发现交易机会→执行买卖决策→管理持仓风险→推送消息。

### 9.1 双引擎架构

| 引擎 | 名称 | 策略 | 运行时段 | 持仓上限 | 单笔金额 |
|------|------|------|----------|---------|---------|
| 引擎1 | 趋势跟踪 | AI信号+复盘推荐+动态板块发现+回踩机会 | 全天 | 5只 | 5000-16000 |
| 引擎2 | 底部发力(IntradayScout) | 刚启动的票：低涨幅+量能放大+加速+板块联动 | 早盘10:00-11:30 | 4只 | 3000-5000 |

**引擎1（趋势跟踪）**是系统的主引擎，对应后文 §13-17 的全部内容——怎么判断大盘→怎么判断板块→怎么决定买卖→仓位风控。以下 §9-12 先介绍盯盘的整体架构、数据来源、启动和扫描流程。

**引擎2（底部发力）**是辅助引擎，策略不同、运行时段不同、风控独立，详见 §18。

两个引擎独立运行，发现候选后汇入同一条买入决策管线（统一因子评估+风控检查）。

### 9.2 管线架构

盯盘系统按管线架构组织，数据单向流动：

```
数据流入→检测发现→情景判断→决策判断→执行下单→风控保护→消息推送
```

### 9.3 Mixin组装

Watcher主控制器通过9个Mixin组装，每个Mixin负责一个功能域：

```
Watcher主控制器（watcher.py）
    ├── MarketStateMixin       — 大盘模式分类+情景引擎+技术拐点+量价背离（market_state.py）
    ├── BuyDecisionMixin       — 买入决策管线（buy_decision.py）
    ├── PositionRiskMixin      — 持仓风控：止损止盈+6级分类+被套分析（position_risk.py）
    ├── SectorContextMixin     — 板块趋势+开盘决策+指数恢复+急涨急跌（sector_context.py）
    ├── AbnormalMonitorMixin   — 异动检测+主动换仓评估（market_anomaly.py）
    ├── IntradayScoutMixin     — 引擎2：底部发力四层筛选（intraday_scout.py）
    ├── ClosingDecisionMixin   — 尾盘决策（late_session.py）
    ├── CloseSummaryMixin      — 收盘快照+持仓报告+信号过期（closeout.py）
    └── DecisionLoggerMixin    — 审计决策日志记录（watcher_decision_logger.py）
```

### 9.4 扫描节奏

| 层级 | 频率 | 做什么 |
|------|------|--------|
| 第一层 | 每轮 ~60秒 | 大盘状态+持仓风控+信号+复盘推荐+引擎2+用户回复 |
| 第二层(市场) | 每3轮 ~3分钟 | 板块趋势更新+全市场快照刷新+异动检测+板块急涨急跌 |
| 第二层(引擎2) | 每3轮 ~3分钟 | 引擎2四层筛选（仅早盘） |
| 第三层 | 每50轮 ~50分钟 | 板块热度排名TOP5+共振标签 |
| 额外 | 每5轮 | 健康检查（30+项） |
| 额外 | 每10轮 | 持仓状态持久化落盘 |
| 额外 | 每12轮 | 回踩机会扫描 |
| 额外 | 每15轮 ~15分钟 | 主动换仓评估（AI异步） |

## 10. 数据从哪来

### 10.1 三条数据连接

盯盘系统从三条通道获取数据：

| 通道 | 连接方式 | 获取内容 | 频率 |
|------|----------|----------|------|
| QMT HTTP | HTTP请求 (192.168.1.33:5000) | 个股实时行情+盘口+逐笔成交+K线+下单 | 每轮on-demand |
| QMT Collector TCP | 长连接TCP (127.0.0.1:15555) | 多指数实时行情+全市场~5000只涨跌幅快照 | 实时推送(约每秒) |
| Telegram Bot API | HTTPS | 推送消息+接收用户回复 | 每轮拉取回复 |

### 10.2 QMT HTTP接口详情

#### 行情类

| 接口 | 返回字段 | 调用时机 |
|------|---------|---------|
| `GET /quotes?codes=a,b,c` | lastPrice/preClose/changePct/high/low/open/amount/askVol[5]/bidVol[5] | 每轮获取监控股票价格 |
| `GET /history/{code}?period=1m&count=240` | 240根1分钟K线(OHLCV) | 启动恢复+计算个股日内指标 |
| `GET /tick/{code}` | 逐笔成交(时间/价格/量/方向) | 评估买入时分析大单流向 |
| `GET /instrument/{code}` | upStopPrice/downStopPrice/totalShares/floatShares | 获取涨跌停价和股本 |

**涨跌停幅度**：600/000/002开头→10%，688/300开头→20%。

#### 交易类

| 接口 | 用途 |
|------|------|
| `POST /order` | 下单（模拟盘） |

### 10.3 QMT Collector详解

#### 它是一个独立进程

`python main.py qmt-collect` → 独立进程，TCP服务端+DB写入器。

#### 工作流程
1. 启动时从QMT回填240根1分钟K线到 `index_snapshots` 表（供盘中重启恢复）
2. 交易时段内(~9:30-15:00)每60秒循环：
   - 调用QMT HTTP获取全市场快照（~5000只涨跌幅+价格+成交额）
   - 调用QMT HTTP获取多指数行情（上证/创业板/科创50）
   - 通过TCP socket推送给所有连接的Watcher
   - 同时写入 `market_snapshots` + `index_snapshots` 表（DB容灾）
3. 收盘后自动退出

#### TCP消息格式（JSON行）

**指数消息：**
```json
{"type":"index","code":"000001.SH","name":"上证指数","price":3350.50,"change_pct":0.35,"pre_close":3338.80,"amount":450000000000,"ts":1715000000.0}
```

**全市场快照：**
```json
{"type":"market","ts":1715000000.0,"stocks":{"000001":{"changePct":1.23,"price":12.50,"amount":50000000},"000002":{...}}}
```

#### Collector自动拉起
盯盘启动时检测端口15555是否被监听。未运行则自动 `subprocess.Popen` 拉起 `python main.py qmt-collect`。

### 10.4 Collector降级与断连保护

- **降级策略**：Collector socket未连接时，从DB `market_snapshots` + `sector_snapshots` 读最近数据，不影响核心盯盘
- **断连检测**：`_check_data_stale()` 每轮检查 `last_db_ts`（最后收到数据的时间戳）
  - 超过3分钟无新数据→标记数据就绪为False→私聊告警→暂停所有买入
  - 数据恢复后→自动标记数据就绪为True→私聊通知恢复→交易重启
- **盘中重启恢复**：从 `index_snapshots` 恢复指数走势，从 `market_snapshots` 恢复最新全市场快照

### 10.5 本地数据库核心表

详见[第23节](#23-数据存在哪存多久)。这里列出盯盘最常用的几张：

**trade_signals（交易信号）** — 最核心，所有买入决策的起点：
- `buy_zone_min/max`：买入区上下限（AI在盘前确定）
- `stop_loss/take_profit/trailing_stop`：出场条件
- `signal_score`：信号评分(0-100)
- `status`：pending→bought/expired/cancelled

**market_snapshots（全市场快照）** — Collector写入，~5000只/轮，含 `change_pct/price/amount`
**index_snapshots（指数快照）** — Collector写入，多指数每轮行情

## 12. 盘中每分钟在干什么

### 12.1 一天的时间表

| 时间 | 阶段 | 做什么 |
|------|------|--------|
| 09:25 | 集合竞价结束 | 第一轮扫描→推送开盘决策汇总（上证+持仓+信号+板块集中度） |
| 09:30-11:30 | 上午连续竞价 | 双引擎全开，正常扫描循环（60秒一轮） |
| 10:00-11:30 | 引擎2活跃 | IntradayScout开始运行（10:00前不运行，防开盘噪音） |
| 11:30-13:00 | 午休 | 休眠等待 |
| 13:00-14:30 | 下午交易 | 引擎1继续运行，引擎2停止 |
| 14:30 | 尾盘触发 | 一次尾盘决策 |
| 14:30-15:00 | 尾盘交易 | 继续扫描但不重复触发尾盘决策 |
| 15:00 | 收盘 | 保存快照+过期信号+推送持仓报告+清理运行时状态+退出 |

### 12.2 每轮扫描的完整步骤

以下是 `_scan()` 方法中按顺序执行的31个步骤。每步独立异常保护，单步失败不阻塞后续步骤：

```

  ├── 从TCP socket读取所有待处理消息
  ├── 去重：跳过DB已恢复的旧数据(ts<=last_db_ts)
  ├── 处理指数消息：更新多指数追踪(上证/创业板/科创50)
  │   └── 去重：连续相同价格不追加(防止污染指标)
  └── 处理全市场快照：更新market_snapshot+市场宽度(涨跌家数)+价格历史

步骤1: 处理已完成异步AI结果
  ├── 追高/二判AI→按板块合并推送(同板块≥3只合并为一条)
  ├── 指数波动AI→附在大盘消息后
  ├── 换仓评估AI→解析JSON→执行卖旧买新
  └── 被套离场AI→每10轮检查一次

步骤2: 检测数据新鲜度
  └── last_db_ts超3分钟→标记数据断连→私聊告警→暂停交易

步骤3: 拉取Telegram用户回复
  └── 解析成交确认/延迟指令→更新信号状态

步骤4: 确定监控范围
  └── 持仓+有pending信号的+复盘推荐的→20-40只

步骤5: 批量获取实时价格
  ├── QMT /quotes批量获取价格+盘口+涨跌停价
  └── 追踪最近10分钟价格历史（用于止跌确认）

步骤6: 更新持仓市值
  └── paper_account.update_prices()→更新每只浮动盈亏

步骤7: 记录基准锚点（仅首轮）
  └── 记录pre_close+QMT涨跌幅→后续健康检查交叉校验用

  ├── 日亏损>3%→触发日内熔断

步骤9: 更新板块趋势（每3轮）
  ├── 从market_snapshot按行业聚合→计算平均涨跌幅+涨跌家数

步骤11: 大盘状态检测
  ├── 提取MicroSignals(7个微观信号)

  ├── classify_market_pattern()→识别16种模式

  └── 更新risk_engine市场环境打分(6维度)

步骤12: 指数技术指标拐点检测
  └── MACD/RSI/KDJ拐点→状态变化时推送一次

步骤13: 持仓止损止盈检查
  ├── 开盘缓冲5分钟：不触发止损(防开盘恐慌扫止损)
  ├── 日内熔断：日亏损>3%→所有浮亏仓位立即平仓(T+1锁定的除外)
  ├── 读risk_level+板块趋势→动态调整触发线

步骤14: 主动退出检查
```

## 13. 怎么判断大盘好不好

大盘判断通过三层协作完成：
1. **ScenarioEngine**（情景引擎）：贝叶斯概率预判
2. **MarketStateMixin**（模式分类）：16种市场模式识别
3. **assess_regime()**（上下文调整）：多维度叠加调整

### 13.1 第一层：情景预测引擎（ScenarioEngine）

前瞻性预判，在模式分类之前运行。这是一个贝叶斯概率状态机。

#### 8种预测情景

| 情景 | 方向 | 含义 | 预设行动 | 触发阈值 |
|------|------|------|----------|---------|
| normal_stable | neutral | 正常平稳 | 正常操作 | ≥50% |
| developing_uptrend | bullish | 正在形成上涨结构 | 回调买入 | ≥40% |
| developing_downtrend | bearish | 正在形成下跌结构 | 收紧止损 | ≥40% |
| accelerating_down | bearish | 下跌加速→可能恐慌 | 暂停买入,减仓 | ≥35% |
| accelerating_up | bullish | 加速上涨→可能冲顶 | 收紧止盈,控仓 | ≥30% |
| potential_reversal_up | bullish | 底部迹象→可能反转 | 准备抄底 | ≥30% |
| potential_reversal_down | bearish | 顶部迹象→可能反转 | 获利了结 | ≥30% |
| dead_bounce | bearish | 弱势反弹/死猫跳 | 趁反弹减仓 | ≥35% |

#### 7个微观信号（MicroSignals，从价格序列提取）

| 信号 | 来源 | 可能取值 |
|------|------|---------|
| price_velocity | 短期价格变化率 | 数值(百分比) |
| price_accel | 速度的变化(加速度) | 数值 |
| ema12_pos | 价格vs日内EMA12 | above/at/below |
| vol_pulse | 量能脉冲 | expanding/contracting/normal |
| breadth_pct | 涨家数占比 | 0~1 |
| rsi_signal | RSI状态 | overbought/oversold/normal |
| macd_signal | MACD方向 | bullish/bearish/neutral |
| bounce_quality | 反弹质量 | strong/weak/none |
| higher_highs/higher_lows | 上升结构 | True/False |
| lower_highs | 下降结构 | True/False |
| range_expanding | 振幅扩大 | True/False |
| vol_price_confirm | 量价配合 | yes/no/neutral |
| breadth_trend | 宽度趋势 | improving/deteriorating/stable |
| ema12_just_crossed | EMA突破 | crossed_up/crossed_down/none |

#### 概率更新机制

每个情景有 `confirm` lambda条件（满足→原始得分+0.15）和 `reject` lambda条件（满足→-0.25）：
```
signal_adj = 1.0 + score
raw_prob = prev_prob × clamp(signal_adj, 0.5, 1.5)
# 归一化到总和1.0
# 未确认的情景×0.92时间衰减，向0.10均值靠近
```

#### 输出：MarketOutlook

| 字段 | 内容 |
|------|------|
| primary | 概率最高的主情景（name/label/probability/direction） |
| alternatives | 次高概率的备选情景列表 |
| key_support/key_resistance | 关键支撑/阻力位 |
| bias | 市场方向（bullish/bearish/neutral） |
| urgency | 紧急程度（critical≥70%/act≥55%/watch≥35%/none<35%）|

#### 情景引擎如何影响决策

在 `assess_regime()` 中叠加：
- 偏空+urgency≥act → risk_level升级一档
- 偏空+urgency≥critical → 强制allow_buy=False
- 偏多+urgency≥act → position_mult×1.2
- 偏多+urgency≥watch → stop_mult×0.9

#### 告警推送时机

在主情景切换/概率大幅变化(>20%)/紧急程度升级/关键关卡接近(<0.3%)时推送 🔮 消息。

### 13.2 第二层：市场模式分类（16种）

#### 允许买入的模式

| 模式 | risk_level | position_mult | stop_mult | entry_rule | 含义 |
|------|-----------|---------------|-----------|------------|------|
| normal | safe | 1.0 | 1.0 | standard | 正常波动,无特殊形态 |
| uptrend | safe | 1.0 | 1.0 | pullback | 持续上行,等回调入场 |
| v_reversal | cautious | 0.5 | 0.8 | confirm | V型反转,需技术确认 |
| w_bottom | cautious | 0.7 | 1.0 | confirm | W型双底,量能验证 |
| gap_down_recover | cautious | 0.5 | 0.8 | confirm | 低开高走,恐慌修复 |
| melt_up | dangerous | 0.3 | 0.7 | pullback | 加速冲顶,小仓追涨 |
| late_rally | dangerous | 0.3 | 0.8 | next_day | 尾盘急拉,次日再看 |
| wide_choppy | dangerous | 0.3 | 1.3 | range_boundary | 宽幅震荡,只做边界 |

#### 禁止买入的模式

| 模式 | risk_level | 触发条件关键词 |
|------|-----------|---------------|
| one_sided | dangerous | 重心持续下移+EMA12下方 |
| inverted_v | dangerous | 冲高后持续回落 |
| panic | extreme | 加速下跌+价格在日内底部 |
| dead_cat | dangerous | 反弹弱+未过50%分位 |
| m_top | dangerous | 两次冲高失败 |
| gap_up_fade | dangerous | 跳空高开后持续回落 |
| late_dump | extreme | 尾盘急剧跳水 |
| fishing_line | extreme | 全天推升尾盘暴跌(钓鱼线出货) |

#### 模式检测方法

多时间窗口滚动对比+日内EMA12/EMA26动态均线+波动率+结构探测：
- **短期窗口（15点≈15分钟）**：捕捉最近动量
- **中期窗口（60点≈60分钟）**：判断盘中趋势
- **多层确认**：每日显著高低点结构探测+追踪

### 13.3 第三层：assess_regime()多维度调整

在模式基础上叠加：

**技术面调整：**
- 价格<MA20且偏离>1%→risk_level升级+position_mult×0.6
- 价格<MA60→risk_level升级+confidence=low
- 多日下跌趋势(连续下跌)→risk_level升级+allow_buy可能被强制关闭

**市场宽度调整：**
- breadth健康度下降→risk_level升级
- 涨跌比极端→升级risk_level

**时间/情景叠加：**
- 尾盘+危险模式→升级（如late_rally risk叠加）
- 情景引擎偏空→升级risk_level
- 情景引擎偏空+critical→强制allow_buy=False

### 13.4 6维度牛熊打分（RiskEngine市场环境）

| 维度 | 数据来源 | 权重 | 正面条件 | 负面条件 |
|------|----------|------|----------|----------|
| 价格vsMA20 | stock_basic表 | ±3 | >+5%→+3 | <-5%→-3 |
| 价格vsMA60 | stock_basic表 | ±2 | >+3%→+2 | <-3%→-2 |
| 量能趋势 | 全市场近5天 | ±2 | 放量>+15%→+2 | 缩量>-15%→-2 |
| 涨跌家数比 | 全市场快照 | ±2 | >2.0→+2 | <0.5→-2 |
| 日内振幅 | 上证high/low | -1 | — | >3%→-1 |
| 活跃板块数 | 板块统计 | ±1 | ≥5→+1 | ≤1→-1 |

总分≥3→bull(仓位80%)，0~2→swing(50%)，<0→bear(20%)。

### 13.5 量价背离检测

最近12个数据点分前后两半比较量价变动：

| 类型 | 条件 | 含义 |
|------|------|------|
| 诱多 | 价涨>0.3%+量缩>15% | 上涨无量，假突破 |
| 恐慌放量 | 价跌>0.3%+量增>15% | 恐慌盘涌出 |

### 13.6 多指数背离检测

追踪上证/创业板/科创50三个指数：

| 场景 | 条件 | 告警 |
|------|------|------|
| 权重护盘 | 上证涨>0.1%+创业板跌>0.5% | 权重护盘小票出货 |
| 创业板高位回落 | 从日高回落>1% | 强势板块回吐 |
| 创业板急跌 | 跌>1% | 科技/成长股回调 |
| 科创50急跌 | 跌>1.5% | 半导体/AI硬件回调 |

### 13.7 技术指标拐点

基于上证分钟级数据计算MACD/RSI/KDJ，检测金叉/死叉/超买/超卖/背离。**只在状态变化时推送一次**。并生成组合信号操作建议（如：超卖+MACD金叉="共振做多信号较强，可分批低吸"）。

### 13.8 AI大盘波动分析

近3轮上证波动≥0.5%时，将分钟级MACD/RSI/KDJ/均线数据发送AI异步分析（不阻塞扫描）：
1. 这波急跌/急拉会继续还是反转？
2. 到什么点位可能企稳？
3. 当前应该追/等/减/守？

AI回复附在大盘波动消息后。AI不可用时用内置简单建议替代。

### 13.9 双重回撤保护

**第一层：日内熔断**
- 日亏损>3%→所有浮亏仓位立即平仓(T+1锁定的除外)
- 每条告警当日只推送一次

**第二层：总账户回撤**
- 总回撤≥15%→阻止新买入+推送清仓建议
- 不影响止损卖出（止损照样执行）

## 14. 怎么判断板块强不强

### 14.1 数据聚合

每3轮（~3分钟），从全市场快照按行业/概念聚合：
- 每个板块计算：平均涨跌幅、涨跌家数、板块总成交额
- 追加到历史序列（`_sector_trend_history` / `_concept_trend_history`）
- 同时计算板块资金量能（当前轮成交额 vs 上轮→量比vol_ratio）

### 14.2 趋势方向判定（线性回归斜率）

用最小二乘法拟合板块均值序列：

| 条件 | 判定 |
|------|------|
| 斜率>+0.003 + 数据点≥5 + 累计>+0.3% | **持续走强** |
| 斜率>+0.003 + 累计≤+0.3% | 走强 |
| 斜率<-0.003 + 数据点≥5 + 累计<-0.3% | **持续走弱** |
| 斜率<-0.003 + 累计≥-0.3% | 走弱 |
| 累计绝对值<0.3% | 横盘 |

斜率阈值±0.003很小，因为板块均值每轮变化通常只有千分之几。n≥5至少15分钟数据，防开盘噪音。

### 14.3 加速度：趋势在加剧还是减缓

| 趋势方向 | 最近半段vs全段 | 加速度 |
|----------|---------------|--------|
| 走强中 | 最近半段>全段×1.5 | 加速 |
| 走弱中 | 最近半段<全段×1.5 | 加速 |
| 走强中 | 最近半段<全段×0.3 | 趋缓 |
| 走弱中 | 最近半段>全段×0.3 | 趋缓 |

### 14.4 相对强度、广度、连续性

- **相对强度**：板块均值-全市场均值。>+0.5%=强于大盘，<-0.5%=弱于大盘
- **板块广度**：(涨家数-跌家数)/总家数。>0.4=普涨，<-0.4=普跌
- **连续性**：追踪连续同方向轮数（`_sector_trend_continuity`）

### 14.5 概念板块判定（独立于行业板块）

取股票所属前5个概念板块，按实时涨跌幅：>+1.0%=强势(+1分)，<-1.0%=弱势(-1分)。综合评分：
- ≤-2：多数概念走弱→买入直接拒绝
- <0：概念偏弱→仓位×0.6

### 14.6 板块共振/逆势分析（SectorResonanceAnalyzer）

大盘波动≥0.3%时自动触发：

四象限分类：
- **共振下行**：板块跟随大盘下跌→弱势，应回避
- **共振上行**：板块跟随大盘上涨→正常联动
- **逆势走强**：抵抗大盘下跌的强势板块→重点关注
- **逆势走弱**：大盘涨但它跌→警惕持仓

输出含：量价标签（放量/缩量）、领涨/领跌个股、趋势起点时间。

### 14.7 板块急涨急跌检测

每3轮检测一次板块级异常：

| 条件 | 告警 | 冷却 |
|------|------|------|
| 板块跌幅>1.5% | 附带持仓标记+涨跌家数+领跌股TOP3 | 全局40轮/单板块90轮 |
| 板块涨幅>2.5% | 附带领涨股TOP3 | 同上 |
| 持仓板块 | 标记🔴 | — |

### 14.8 在买入决策中的四级影响

| 板块趋势 | 区内买入 | 回调买入 | 仓位 | 高于买入区 |
|----------|----------|----------|------|-----------|
| 持续走弱 | 🚫拒绝 | 🚫watching | ×0.3 | 跳过 |
| 走弱 | ⚠️×0.5 | -1分 | ×0.6 | 跳过 |
| 横盘 | 不影响 | 不影响 | 不影响 | 跳过 |
| 走强 | 不影响 | +1分 | ×1.2 | 跳过 |
| 持续走强 | ✅×1.2 | +3分 | ×1.3 | 📈追高提醒 |

**核心原则：板块持续走弱时坚决不逆势买入；板块持续走强时顺势加仓。**

## 15. 怎么决定买不买

买入决策管线是系统最复杂的部分。所有候选（来自4个渠道）汇入统一管线处理。

### 15.1 四个候选来源

| 渠道 | 来源 | 触发 | 买入区间 |
|------|------|------|----------|
| AI信号 | trade_signals表，status=pending | 每轮 | AI盘前确定（精确限价） |
| 复盘推荐 | stock_tracker表，来源='复盘' | 每轮 | 动态：MA10±1%或MA20-1%/+2% |
| 动态板块发现 | 盘中实时计算热门板块龙头 | 每3轮 | 系统动态计算 |
| 回踩机会 | 全市场扫描：强势板块+回踩支撑 | 每12轮 | 基于支撑位动态计算 |

### 15.2 统一候选处理管线

所有候选进入 `_check_buy_candidates()`，按当前价格分三种情况：

```
情况1: price > buy_max（高于买入区）
  ├── 板块持续走强→📈推送追高提醒+AI异步二判
  ├── 距上限≤3%+市场偏空→📉买入区接近预告
  └── 否则→跳过

情况2: buy_min ≤ price ≤ buy_max（在买入区内）
  ├── 防重复推送检查
  ├── 大盘不允许→⏸️搁置推送
  ├── 涨停了→🚫涨停无法买入
  ├── entry_rule过滤（见15.4节）
  ├── 动态买入区修正（见15.5节）
  ├── 做9因子评估→
  │   ├── reject→⛔跳过+AI异步二判
  │   ├── warn→⚠️减仓执行
  │   └── 通过→🔴推送+模拟盘执行
  └── 满仓→推送信号但不执行

情况3: price < buy_min（低于买入区）
  └── 做8因子评分→
      ├── < -4分→❌放弃(expired)
      ├── -4~2分→👀继续观察
      └── ≥3分→🟢回调买入
```

### 15.3 区内买入的10个评估因子（evaluate_buy）

逐层检查，返回(是否允许, 原因, 仓位系数)。仓位系数从1.0开始，每层可能调整：

**因子1：行业板块趋势**
- 持续走弱/板块数据不足→🚫拒绝
- 走弱→⚠️×0.5
- 持续走强→✅×1.2

**因子2：买入区位置**
- zone_pos≥85%→🚫拒绝(不追高)
- ≥65%→⚠️×0.7
- ≤33%→×1.1(激进)

**因子3：布林带+均线关系**
- %B≥90→🚫拒绝
- ≥75→⚠️×0.8
- 空头排列+破位→🚫拒绝(接飞刀)
- 价格<所有均线(非标准空头)→⚠️×0.7
- 回踩关键支撑(下轨或MA20±3%)→×1.2

**因子4：日线KDJ/RSI**
- RSI6≥85→🚫拒绝
- ≥75→⚠️×0.7
- KDJ J>100→🚫拒绝
- >85→⚠️×0.7

**因子5：分钟级RSI/MACD/KDJ（QMT实时K线计算）**

- KDJ J>100→🚫拒绝；>85→⚠️×0.7；<0→×1.1
- KDJ死叉→⚠️×0.8
- 价格<日内MA5超过3%→🚫拒绝(急跌接飞刀)

**因子6：盘口买卖力量**
- 买盘占比≤30%→🚫拒绝；≤42%→⚠️×0.85；≥70%→×1.1

**因子7：大单流向**
- 大单买入占比≤35%→🚫拒绝；≤45%→⚠️×0.8；≥65%→×1.1

**因子8：涨跌停空间**
- 距涨停<2%→🚫拒绝；<4%→⚠️×0.8

**因子9：昨日趋势背景**
- 主力净流入<-5%→🚫拒绝；-5%~-2%→⚠️×0.85
- MA5角度<-2→🚫拒绝；-2~0→⚠️×0.85
- 日线KDJ J>100→🚫拒绝；>85→⚠️×0.8
- 日线MACD空头+bar<-0.3→⚠️×0.85

**因子10：今日实时因子+价格走势**
- 日内位置<15%+MA5向上→×1.1
- 价格>日内90%→⚠️×0.85
- 5分钟MACD多头→×1.05；空头+bar<-0.2→⚠️×0.85

| entry_rule | 含义 | zone_pos过滤 |

### 15.5 动态买入区修正

`calc_dynamic_buy_zone()`根据三层联动因子(大盘→板块→个股)评估是否下移买入区：
- 修正幅度<2%→静默使用不告警
- 修正幅度≥2%→推送🎯告警
- 下移范围：0~15%

### 15.6 回调买入的8因子评分

当价格<买入区下限时，打分而不是简单放弃：

| 因子 | 加分条件 | 扣分条件 |
|------|----------|----------|
| 偏离买入区幅度 | ≤2%→+2 | >7%→-5 |
| 关键支撑距离 | 触及下轨/MA20/MA60±2-3%→+4 | 跌破MA60→-3 |
| 日内技术 | RSI6≤25→+3;KDJ低位金叉→+2 | MACD空头→-2 |
| 大单方向 | 买入≥60%→+3 | 卖出≥60%→-3 |
| 盘口支撑 | 买盘≥65%→+2 | 卖盘≥65%→-2 |
| 板块趋势 | 持续走强→+3 | 持续走弱→直接watching |
| 量能验证 | 缩量下跌(vol_ratio<0.5)→+2 | 放量下跌(>2)→-2 |
| 昨日+今日 | 主力流入+MA5向上→+1~3 | 主力流出+加速下行→-1~-3 |

总分≥6→opportunity(满仓)，3~5→opportunity(0.7×)，0~2→watching，-4~-1→watching，<-4→abandon。

### 15.7 AI二判机制

当系统拒绝买入时，通过**AI异步队列**提交二判请求（不阻塞扫描）：
- 追高场景：AI判断是否值得追
- 被拒场景：AI二判拒绝是否合理
- 结果异步返回→推送到TG
- 同板块≥3只追高提醒→合并为一条推送

### 16.1 仓位计算五步骤

|----------|---------|

**步骤2：市场宽度修正**

**步骤3：板块趋势修正**

**步骤4：AI板块倾向修正**
focus→×size_multiplier(1.0~1.3)，avoid→跳过

下沿33%→×1.1，上沿67%→×0.7

最终：百元取整，下限5000，上限16000。

### 16.2 风控引擎（RiskEngine）检查

```python
1. 黑名单检查 → is_blacklisted(code) 查 stock_risk_pool 表
2. MarketRegime仓位限制 → position_mult≤0→拒绝
3. 市场环境仓位上限 → bull 80%/swing 50%/bear 20%
4. 集中度检查:
   ├── 单只≤20%总资产
   ├── 单板块≤70%总资产
   └── 现金保留≥20%
5. 最大回撤检查 → 日亏损>3%→熔断
```

### 16.3 统一止损止盈计算

**动态止损（calc_unified_stop_loss）**：
```
止损宽度 = ATR(14) × 2.0  ← 基础宽度
  × 策略修正：趋势1.2× / 追涨0.8×
  × 板块修正：走强1.1× / 走弱0.85×
止损价 = max(入场价 - 止损宽度, 入场价 × 0.93)  ← 93%硬地板
止损价 = max(上述, 下方支撑位)  ← 支撑位约束
```

**动态止盈（calc_unified_take_profit）**：
```
止盈宽度 = ATR(14) × 3.0
  × 策略修正：趋势1.3× / 追涨0.7×
  × 板块修正
止盈价 = min(入场价 + 止盈宽度, 入场价 × 1.12)  ← 112%硬天花板
止盈价 = min(上述, 上方阻力位)  ← 阻力位约束
```

买入时根据MarketRegime的stop_mult调整止损宽度：`effective_sl = price - (price - sl) × stop_mult`。

## 17. 怎么决定卖不卖

### 17.1 六个卖出触发条件

**类型1：固定止损**
- 触发：现价≤止损价
- T+1保护+开盘5分钟缓冲

**类型2：固定止盈**
- 触发：现价≥止盈价

**类型3：移动止盈**

**类型4：利润回撤止盈（三级）**
- T1(最高浮盈≥15%)：保留60%→当前<最高×0.60触发
- T2(≥10%)：保留55%
- T3(≥5%)：保留50%
- 大盘极端/危险时bonus+10%（更保守）

**类型5：时间止损**
- 触发：持有>5天且亏损>7%

**类型6：日内熔断强制平仓**
- 触发：日亏损>3%→所有浮亏仓位立即平仓

### 17.2 大盘/板块感知的动态调整

止损/止盈/移动止盈触发线根据risk_level动态调整：

| dangerous | 0.85 | 0.90 | 0.85 |

## 18. 盘中底部发力（引擎2）

### 18.1 设计理念

找「刚要涨」的票，不是「已经涨了」的票。四个条件：低涨幅+量能放大+涨幅加速+板块联动。

### 18.2 四层筛选管线

#### Layer1：硬条件筛选
- 涨幅范围：0.5%~3.5%（刚启动，还没涨太多）
- 量能环比放大：>1.2×（有资金进来）
- 涨幅加速：当前涨幅-上轮涨幅>0.3%（在加速不是减速）
- 板块排名：所属板块在TOP10且涨幅>0%
- 股价>5元（排除垃圾股）
- 排除：放量下跌形态、高位加速、近60轮卖出的冷却期股票
- 大盘极端/危险→暂停
- 日亏损>2%→暂停

#### Layer2：六维评分排序（满分100）
| 维度 | 满分 | 评估内容 |
|------|------|---------|
| 加速强度 | 30 | 涨幅加速度的大小和持续性 |
| 量能爆发 | 22 | 量比大小、放量速度 |
| 板块强度 | 18 | 板块排名、板块涨幅、板块内龙头地位 |
| 位置安全 | 18 | 距均线距离、距高低点位置 |
| 趋势质量 | 12 | 均线方向、MACD方向 |
| 大盘配合 | 10 | 市场环境是否有利于底部发力 |

总分排序→取前N进入Layer3。

#### Layer3：AI异步过滤
- 使用breakout场景模板→判断突破有效性
- 系统原则：真正强势的股票不会盘中回踩；尾盘14:30后仓位减半
- AI超时45秒→跳过该候选

#### Layer4：结果处理+买入执行
- AI判断通过→调用统一买入管线执行
- AI判断不通过→推送关注提醒

## 19. 异动检测和换仓

### 19.1 异动检测（每3轮，每50轮一次综合）

**个股异动（AbnormalDetector）：**
| 类型 | 阈值 | 含义 |
|------|------|------|
| 急速拉升 | 涨幅变化>1% | 盘中突然拉涨 |
| 量比暴增 | 量比>3× | 成交量异常放大 |
| 逼近涨停 | 涨幅>8.5% | 接近涨停板 |

**板块异动（detect_sector）：**
- 同行业≥3只同时拉升→板块级异动告警

**一句话行情判断：**
`format_market_judgment()`综合多指数+宽度输出：普跌/创指失真/权重护盘/共振下跌等。

### 19.2 板块热度排名（每50轮）

SectorHeatMonitor基于全市场快照计算TOP5板块：
- 各板块平均涨跌幅排序（最少3只股票）
- 整合共振分析标签：共振下跌/逆势上涨/放量/缩量
- 格式化推送至TG

### 19.3 主动换仓评估（每15轮）

当持仓≥3只且pending信号在买入区附近时触发：

**流程：**
1. 收集候选：pending信号中筛出在买入区内或接近的
2. 构建上下文：大盘+板块+持仓涨跌+候选涨跌
3. **AI异步评估**（不阻塞扫描）→判断是否换仓→返回JSON `{"sell":"code","buy":"code"}`
4. AI不可用时fallback：优先卖close建议→其次reduce→再次评分差距>15分
5. 执行：先卖后买，T+1保护

**不触发换仓的情况：**
- 持仓<3只
- 无候选在买入区附近
- AI决定不换
- 卖出标的当日买入（T+1锁定）

## 20. 尾盘怎么处理

14:30后触发一次，逐只判断是否持过夜：

| 情况 | 建议 | 触发条件 |
|------|------|----------|
| 今日买入 | 🔒不可卖 | T+1锁定 |
| 大盘极端+浮亏 | 🚨建议清仓 | risk_level=extreme+浮亏 |
| 浮亏>3% | 🔴建议止损 | 不持亏过夜 |
| 浮亏1-3%+距SL<3% | 🟡关注 | 可能触发止损 |
| 浮盈>5%+大盘危险 | 🟢建议减仓 | 利润可观+大盘风险 |
| 其他 | ✅可持过夜 | — |

仅在需要行动时才推送消息，每只附带板块趋势。大盘极端时标题加⚠️标记。

# 第五部分：消息与审计

## 21. 消息系统

### 21.1 推送架构

- **群聊**（TELEGRAM_CHAT_ID）：所有盯盘消息、复盘报告、盘前简报、预测核验
- **私聊**（TELEGRAM_PRIVATE_CHAT_ID）：实盘止损止盈确认、健康检查告警、数据断连告警
- **AlertRouter**：统一路由层，双通道，指纹去重，代码级冷却

### 21.2 去重机制

AlertRouter支持两种去重：
- **指纹去重**：按消息首行提取股票代码或前40字符，N轮内同指纹不重复
- **代码级冷却**：同一股票代码在7轮内且价格变化<0.5%→抑制推送

### 21.3 消息类型速查

**大盘类（~11种）**：
🚨恐慌下跌/单边下跌/大盘熔断 → ⚠️弱势反弹/冲高回落/尾盘跳水 → 🔄V型反转 → ⚡急拉急跌+AI分析 → 🚨最大回撤+日内熔断 → 🔮情景预判 → ⚠️多指数背离 → ⚠️板块急涨急跌

**指数技术拐点（~12种组合）**：
📈 MACD金叉/死叉+RSI超买/超卖+KDJ极值+背离+操作建议

**买入类（~7种）**：
🔴买入信号+9因子分析 → 🟢回调买入+评分详情 → ❌信号放弃 → ⏸️大盘搁置 → 🚫涨停无法买入 → 📈追高提醒+AI二判 → 📉买入区接近预告

**卖出/持仓类（~10种）**：
⚠️止损/✅止盈/⚠️移动止盈/利润回撤触发 → 6级状态分类 → ⚠️接近告警 → 🎯动态修正 → 🔴被套分析(AI) → 🟡补仓机会

**模拟盘交易（2种）**：
📝模拟盘买入+盈亏预期 → ✅⚠️模拟盘卖出+实际盈亏

**汇总类（4种）**：
📋开盘决策汇总 → 📊板块热度TOP5 → 📊收盘持仓报告(模拟→群/实盘→私) → 📊预测核验报告

**提醒类（3种）**：
⏰止损再提醒(5分钟循环) → 🔔尾盘决策 → 私聊确认请求

### 21.4 Telegram消息接收

`MessageReceiver`通过getUpdates长轮询接收用户消息，写入`storage/telegram_last_update_id.txt`防重复。

`ManualExecutor`解析用户的自然语言回复：
- "成交002371"→更新信号bought+记录订单
- "未成交002371"→标记rejected
- "再等10 002371"→延迟提醒10分钟
- 支持中英文混合、分段匹配

## 22. 策略与盯盘审计

### 22.1 两个审计命令

**`strategy-audit`** — 策略管线审计：
```
python main.py strategy-audit [日期] [--apply N] [--list]
```

**`audit`** — 统一审计：
```
python main.py audit [--domain watcher|strategy] [--rule-only|--ai-only] [--apply N] [--list]
```

### 22.2 双轨审计机制

每条审计都包含规则审计+AI审计两轨：

**规则审计（RuleAuditor）**：纯统计，不调AI
- 策略审计：因子胜率/因子交互/阈值敏感性/skip反事实分析/场景表现
- 盯盘审计：市场模式分类验证/买入信号质量/止损质量/止盈质量/仓位大小合理性/板块共振分析

**AI审计（AIAuditor）**：把规则审计发现+历史决策数据发给AI（用AI_MODEL_AUDIT模型）
- 偏见检测/遗漏模式/改进建议（具体到参数和代码修改）

### 22.3 盯盘决策日志

`DecisionLoggerMixin`记录每笔决策到`watcher_decision_log`表：
- 买入/卖出/换仓/追高/回调等所有决策类型
- 记录决策时的完整上下文（大盘状态/板块趋势/价格/评分）
- 供事后审计分析

### 22.4 改进闭环

1. 审计→发现写入`audit_findings`+`watcher_lessons`/`strategy_lessons`
2. 生成改进建议→`watcher_improvements`/`strategy_improvements`(status=pending)
3. `--apply N`→自动应用（记录applied_date）
4. 后续跟踪`effectiveness_check`

| 止盈修正触发 | <原×0.97 | 下调≥3%→告警 |
| 止损收紧触发 | >原×1.02 | 收紧>2%→告警 |
| 买入区下移修正 | ≥2% | 告警；<2%静默 |
| 预测告警去重 | 15轮 | 同目标连续间隔 |
| 动态修正去重 | 20轮 | 同持仓间隔 |
| SL收紧下限 | max(0.5,2.0-sl_tighten) | 最小宽度 |
| 支撑地板保护 | ×0.99 | 不低于支撑 |

## 28. 已知问题

1. **DNS绕过临时禁用**（2026-06-08）：`dns_bypass.py` 的全局 socket patch 可能导致 curl_cffi 连接异常，当前已注释掉，待观察
2. **Collector断连处理**：Collector crash 时 Watcher 降级到DB模式运行；Collector 自动拉起依赖 subprocess.Popen，若 Python 环境路径不对可能失败
3. **数据库单点**：SQLite 不支持并发写入，monitor 和 listen 同时运行时需确保不同时写同表
4. **QMT /all_quotes 接口 amount 字段**：部分股票 amount 可能为 0 或异常值，板块资金量能计算依赖此字段
5. **Batch API稳定性**：DashScope Batch API 延迟可能超过 BATCH_TIMEOUT_MINUTES 配置（默认60分钟）
6. **代理池整点刷新**：代理在整点可能集体失效，采集器有120秒重试机制但批量采集时可能部分失败
7. **PID文件残留**：非正常退出时 storage/watcher.pid 可能残留，需手动删除后重启（系统已处理僵尸PID清理）
8. **多指数数据依赖Collector**：创业板/科创50指数数据完全依赖 Collector TCP 推送，HTTP 通道仅获取上证
