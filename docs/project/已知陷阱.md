# 已知陷阱

踩过的坑，免的再踩。每个陷阱记清楚现象、根因、解法。

---

## QMT 行情相关

- **QMT HTTP 卡死 Watcher**：`self.qmt.get_realtime()` 的 `READ_TIMEOUT=120s`，网络抖动可让 Watcher 僵死 2 分钟。个股实时价必须从 `_market_snapshot`（Collector TCP 推送）取，QMT HTTP 仅用于补充涨跌停价（`preClose`）
- **不偷跑**：2026-06-10 全天"无实时行情"，Collector 连接正常但 `_market_snapshot` 为空时（开盘初期或网络断），`_scan()` 跳过整轮。可接受，不应绕过
- **`/quotes` 不含 `preClose`**：QMT 批量行情接口不返回昨日收盘价，需单独 `/quote/{code}` 获取
- **QMT 只用于行情**：下单功能不可用，模拟盘本地记账
- **QMT 代码后缀**：`QuoteClient` 自动处理 .SH/.SZ，手动拼接会导致查不到数据
- **QMT Collector 自动拉起**：`_ensure_collector_running()` 检测 15555 端口，不在则 `subprocess.Popen` 拉起。Watcher 退出时 Collector 不会自动停

## Watcher 稳定性

- **进程静默卡死**：2026-06-10 11:25 后进程无日志、0% CPU、S 状态。原因待排查，疑似 `self.qmt.get_realtime()` HTTP 阻塞。临时方案：kill 重启
- **`_get_realtime_prices` 在 `_scan()` 中**：返回空则整轮跳过，所有买入/卖出逻辑不执行
- **`build_state` 引用传递**：传的是列表/字典引用而非拷贝，调用方**不得**直接修改 dict/list 内容

## 数据一致性

- **`index_close_low` 初始化**：启动后首轮 Collector 数据未到时值为 0，健康检查 fallback 到 K 线高低点数据源（`index_low`），与收盘价序列（`index_prices`）不一致导致误报"最低价不一致"。已加非零保护：`if ctx.index_close_high and ctx.index_close_low:`
- **`index_snapshots` 存两套数据**：`high`/`low` 是 K 线高低点，`price` 是收盘价。`_restore_index_context` 分别恢复，`index_low` = min(K线low)，`index_close_low` = min(收盘价)。两者天然不同，不能互比
- **`insert_snapshot` 用 `INSERT OR REPLACE`**：UNIQUE(trade_date, account)，重复插入不会报错但会覆盖旧值
- **`stock_basic` schema 以生产库为准**：`schema.py` 中 CREATE TABLE 仅作文档用，生产库由采集器 INSERT 自动创建列（SQLite 特性）。新功能加字段走 `ALTER TABLE`，不能依赖 schema.py

## 代码路径陷阱

- **`repo.get_money_flow` 的 import 路径**：`data/repo/__init__.py` 中实际 import `data.market.stock_basic.StockReader`，别被 `from data.readers.stock_reader` 误导（这是旧的注册表路径）
- **`stock_tracker` 字段名**：评分字段是 `star_rating`，不是 `score`
- **`AIQueue` 已委托给 `system.ai`**：不再自己管理线程，不要 `self._ai_queue = AIQueue()` 后单独 start
- **`trade/scenario/templates/` 已删除**：场景模板迁移到 `system/ai/prompts/`，import 旧路径会报错
- **CLS API 已迁移**：`/nodeapi/telegraphList` → `/api/cache?name=telegraph`

## 网络/环境

- **`dns_bypass`**：绕过小火箭 DNS 劫持，DNS 解析相关代码必须走这个
- **`telegram.py` 中 `requests verify=False`**：因为小火箭 HTTPS MITM，不加这个 Telegram API 连不上
- **FunctionCalling 工具注册**：新增工具要同时改 `system/ai/stock_tools.py`（TOOLS_DEFINITION）和 `function_calling.py`（tool_functions），漏一个 AI 就调不到

## macOS 环境

- **tesseract 沙箱无法读 /tmp**：Homebrew 安装的 tesseract 不能访问 `/tmp`（符号链接到 `private/tmp`），报错 `failed to open locally with tail...`。OCR 临时文件必须写到项目目录（如 `storage/pdf/`）或 tesseract 有权访问的路径。`regulatory_analysis.py` 的 `_ocr_pages()` 已处理此问题（work_dir 用 PDF 所在目录）

## 交易逻辑陷阱

- **模拟盘止损是自动的**：`_handle_stop_signal` 中 `execute_paper_sell` 直接执行，仅在跌停封单（`_is_limit_down`）时跳过。不会等用户确认
- **T+1 锁定**：`_check_positions` 中当天买入的仓位不触发止损止盈（`is_today_buy`），第二天才放开
- **深跌等待反弹**：亏损 > 7% 且不是当日买入 → 进入深跌状态机，不立即止损，等反弹失败或 14:00 后才卖
- **市场宽度仅拦盘前信号**：`_check_buy_candidates` 中宽度过滤只对 `source in ("signal", "review")` 生效，盘中动态发现和回踩扫描不拦
- **`entry_rule="standard"` 绕过 zone_pos 过滤**：当 market pattern 从 "normal" 变成 "unknown" 时，entry_rule 从 "confirm" 变成 "standard"，原代码中 standard 没有 zone_pos 检查。结果：被 "confirm" 过滤的股票在 pattern 切换后裸奔通过。已修复：standard 加了 zone_pos>0.5→暂缓
