# 第一部分：系统全景

## 1. 系统全景图

### 1.1 一句话概括

这是一个 **A 股量化交易辅助系统**。它运行在 macOS 上，通过网络连接到 Windows 电脑上的 QMT（券商量化终端），每天自动完成以下工作：

- **盘后**：采集全市场数据 → AI 分析复盘 → 策略筛选候选股 → 生成买入信号
- **盘前**：汇总隔夜资讯 + 宏观变化 → AI 生成盘前简报
- **盘中**：盯盘扫描 → 判断大盘/板块 → 触发买入信号 → 风控止损止盈 → 模拟盘自动交易
- **事后**：审计策略表现 → 发现决策偏差 → 提出改进建议

所有消息通过 Telegram 推送到你手机上，你也可以通过 Telegram 回复来控制实盘操作。

### 1.2 系统由哪些部分组成

整个系统可以理解为 **7 个子系统** 拼在一起：

```
┌─────────────────────────────────────────────────────┐
│                  量化交易系统                        │
├───────────────┬─────────────────┬──────────────────┤
│  数据采集      │   分析决策       │   交易执行        │
│               │                 │                  │
│ • 16个采集器   │ • 盘后复盘管线   │ • 盯盘扫描        │
│ • 代理池管理   │ • 盘前简报       │ • 买入决策        │
│ • QMT Collector│ • 策略选股管线   │ • 止损止盈        │
│               │ • 策略审计       │ • 模拟/实盘       │
│               │ • 个股追踪       │ • 换仓评估        │
├───────────────┴─────────────────┴──────────────────┤
│  消息层: Telegram Bot（推送 + 接收用户回复）          │
│  存储层: SQLite（42 张表，共用 stock_market.db）     │
└─────────────────────────────────────────────────────┘
```

每个子系统通过 `python main.py <命令>` 启动，由 cron 定时调度或手动触发。

### 1.3 七个命令，七个子系统

| 命令 | 子系统 | 做什么 | 触发方式 |
|------|--------|--------|----------|
| `review` | 盘后复盘 | 采集数据 → AI 写复盘报告 → 触发策略管线 → 推送 Telegram | cron 每天 18:00 |
| `morning` | 盘前简报 | 收集隔夜资讯+宏观+电报 → AI 生成简报 → 推送 Telegram | cron 每天 08:30 |
| `strategy` | 策略选股 | 市场宽度 → 趋势筛选 → AI 研判 → 生成买入信号入库 | review 成功自动触发 |
| `monitor` | 盯盘交易 | 盘中每分钟扫描，判断买卖，执行模拟盘，发 Telegram | cron 每天 09:25 |
| `listen` | 消息监听 | 长轮询 Telegram，接收你的回复（成交确认/延迟指令） | cron 每天 09:00-18:00 |
| `collect` | 数据采集 | 16 个采集器获取全市场数据（可按模块过滤） | 部分 cron 每 5 分钟 |
| `strategy-audit` | 策略审计 | 规则审计+AI审计→发现决策偏差→生成改进建议 | 手动 |

还有几个辅助命令：`track`（个股追踪）、`compare`（模拟vs实盘比对）、`cleanup`（数据清理）、`qmt-collect`（QMT 实时行情中转）、`test`（配置检查）、`portfolio`（持仓查询）。

### 1.4 关键外部依赖

| 组件 | 在哪 | 干什么 |
|------|------|--------|

| SQLite | macOS 本地 (storage/stock_market.db) | 所有数据存储 |
| DeepSeek / Qwen | 云端 API | AI 分析（复盘、选股、审计、波动分析） |
| 东方财富 API | 公网 | 全市场行情、板块、龙虎榜等数据 |
| 财联社 | 公网 | 实时电报、早报 |
| 代理池 | 公网 | 反爬代理轮换 |

## 2. 一天的时间线

系统按照交易日的自然节律运行，每个时段有明确的任务：

```
08:00 ─ 电报采集开始（每 20 分钟一次，持续到 20:00）
08:30 ─ 盘前简报：汇总隔夜宏观 + 电报 + 早报 → AI 写简报 → 推 Telegram
09:00 ─ Telegram 监听启动（listen 进程）
09:25 ─ 盯盘系统启动（monitor 进程），集合竞价结束第一轮扫描
09:30-11:30 ─ 上午连续竞价，盯盘每分钟扫描
11:30-13:00 ─ 午休，盯盘休眠
13:00-14:30 ─ 下午连续竞价
14:30 ─ 尾盘决策：逐只判断是否持过夜
15:00 ─ 收盘：保存快照、过期信号、推送持仓报告 → 盯盘 + 监听退出
18:00 ─ 盘后复盘：采集全市场数据 → AI 分析写报告 → 推 Telegram
       └─ 复盘成功后自动触发策略管线：
          市场宽度 → 趋势筛选 → 因子评分 → AI 研判 → 信号入库
20:00 ─ 电报采集停止
```

## 3. 数据采集系统

### 3.1 它是干嘛的

数据采集系统负责从互联网获取 A 股全市场数据，存到本地 SQLite，供复盘、选股、盯盘使用。通过 `python main.py collect` 启动，支持按模块筛选。

### 3.2 16 个采集器一览

| 分类 | 采集器 | 数据源 | 写入表 | 采集内容 |
|------|--------|--------|--------|----------|
| 行情 | StockBasicCollector | 东方财富 | stock_basic | 全市场 ~5000 只股票日线（价格/量能/资金/市值/行业） |
| 行情 | MainIndexCollector | 东方财富 | index_realtime_data | 上证/深证/创业板等主要指数 |
| 行情 | IndustryBoardCollector | 东方财富 | sector_industry | 申万行业板块行情 |
| 行情 | ConceptBoardCollector | 东方财富 | sector_concept | 概念板块行情 |
| 行情 | SectorStocksCollector | 东方财富 | sector_stocks | 板块→成分股映射 |
| 行情 | SuspendResumeCollector | 东方财富 | stock_suspend_resume | 停复牌信息 |
| 事件 | LimitPoolCollector | 东方财富 | limit_pool | 每日涨跌停池 |
| 事件 | StrongStockCollector | 东方财富 | strong_stock | 强势股（连续涨停等） |
| 事件 | LHBCollector | 东方财富 | lhb_stocks / lhb_seats | 龙虎榜上榜股票 + 席位明细 |
| 事件 | StockMonitorCollector | 东方财富 | stock_monitor | 个股异动监控 |
| 事件 | RegulatoryLetterCollector | 东方财富 | regulatory_letter | 监管函件 |
| 事件 | ShareHolderChangeCollector | 东方财富 | share_holder_change | 股东增减持 |
| 事件 | NoticeCollector | 东方财富 | future_announcements | 未来公告预告（业绩/分红/解禁） |
| 资讯 | TelegraphCollector | 财联社 API | cls_telegraph | 实时电报（盘中每 20 分钟） |
| 资讯 | CLSDigestCollector | 财联社 | — | 早报/晚报（用于复盘和盘前） |
| 宏观 | MacroCollector | 东方财富 | macro_daily | Shibor/国债/汇率/北向资金等 |

### 3.3 代理与反爬

数据采集走代理池，核心机制：
- **ProxyManager**：管理代理 IP 池，定时验证可用性
- **ProxyRequester**：封装 requests，自动重试、换 IP
- **DNS 绕过**：`dns_bypass.py` 在系统启动时 patch Python socket 层，绕过 Surge/Clash 等本地代理工具的 DNS 劫持

不是所有采集器都走代理。财联社 API（TelegraphCollector）直连，不走代理。

### 3.4 采集器怎么运行

三种运行方式：

| 方式 | 命令 | 时机 |
|------|------|------|
| 复盘全量采集 | `review` 命令内部调用 13 个采集器 | 每天 18:00 |
| 电报定时采集 | `start_news_collection.sh` → `collect --module news` | 每 20 分钟 |
| 手动指定模块 | `python main.py collect --module market` | 按需 |

`review` 命令的 `ReviewService.collect()` 会依次调用 13 个采集器，每个失败后会等 120 秒重试一次（应对代理池整点刷新）。

## 4. 盘后复盘管线

### 4.1 它是干嘛的

每天收盘后，自动采集全市场数据 → 清洗统计 → AI 写分析报告 → 推送到 Telegram。复盘成功后自动触发策略选股管线。

启动：`python main.py review`（cron 每天 18:00）

### 4.2 完整流程

```
Step 1: 数据采集（13 个采集器依次跑）
  ├── 行业板块 / 概念板块
  ├── 个股行情（~5000 只）
  ├── 强势股 / 龙虎榜 / 涨跌停池
  ├── 大盘指数 / 监管函 / 停复牌 / 股东增减持
  ├── 公告预告 / 隔夜宏观
  └── 早报摘要（CLSDigest）

Step 2: 数据统计（CollectionStatsService）
  ├── 采集完成度检查
  ├── 板块数据完整性验证
  └── 输出统计报告

Step 3: AI 分析（ReviewAnalyzer）
  ├── 读取各类数据，组装分析 Prompt
  ├── 调用 AI（Qwen/DeepSeek）生成复盘报告
  ├── Function Calling 辅助：查股票代码、查板块、查龙虎榜等
  └── 报告保存到 storage/reports/

Step 4: 其他处理
  ├── 涨停股次日表现分析（ZTPerformanceProcessor）
  ├── 板块采集进度记录（SectorInfoProcessor）
  └── 选股跟踪更新（ StockTracker）

Step 5: Telegram 推送
  └── 消息发送到群聊

Step 6: 自动触发策略管线
  └── cmd_strategy()
```

### 4.3 AI 怎么写报告

复盘 AI 会收到一份结构化的数据 Prompt，包含：
- 今日大盘概况（涨跌幅、成交量、涨跌家数、振幅）
- 板块轮动（领涨/领跌板块 TOP10、板块持续性）
- 涨停分析（涨停家数、炸板率、连板高度、龙头股）
- 龙虎榜（机构/游资动向、席位买卖）
- 资金流向（主力净流入/流出板块、北向资金）
- 异动个股（量比暴增、急速拉升、监管关注）
- 宏观变化（Shibor/汇率/外围市场）

AI 基于这些数据生成：大盘研判、板块机会、个股梳理、明日关注。

## 5. 策略选股管线

### 5.1 它是干嘛的

盘后复盘完成后自动运行（也可手动 `python main.py strategy`）。从全市场 ~5000 只股票中，经过多层筛选 + AI 研判，最终产出可交易的买入信号，写入 `trade_signals` 表供次日盯盘使用。

### 5.2 筛选漏斗

```
全市场 ~5000 只
    │
    ▼ 第 1 层：市场宽度过滤（MarketBreadth）
    │   ├── 涨跌家数 < 熊市阈值？→ 不生成新信号
    │   ├── 恐慌后反弹？→ 只做反弹模式
    │   └── 正常 → 继续
    │
    ▼ 第 2 层：趋势筛选（TrendScreener）
    │   ├── 价格站上 MA20？MA5 角度向上？
    │   ├── 成交额 > 1 亿？市值 > 50 亿？
    │   ├── 板块趋势过滤：板块走弱 → 排除
    │   └── 通过 → 候选池（通常 50-200 只）
    │
    ▼ 第 3 层：因子评分（ProfileBuilder）
    │   ├── 技术面：MACD/RSI/KDJ/布林带/均线
    │   ├── 资金面：主力净流入/量比/换手率
    │   ├── 位置面：距 52 周高低点/距 MA60
    │   └── 生成 StockProfile（含评分 + 场景分类）
    │
    ▼ 第 4 层：AI 研判（AIAdvisor）
    │   每个候选经过 AI 深度分析：
    │   ├── 我看到了什么？（what_i_see）
    │   ├── 我担心什么？（what_concerns_me）
    │   ├── 决定性因素？（decisive_factor）
    │   ├── 判定：入选 / 淘汰
    │   ├── 如果入选：买入区上下限、止损价、止盈价
    │   └── 如果淘汰：原因、什么条件会重新考虑
    │
    ▼ 入库
        ├── strategy_funnel：记录每层通过情况
        ├── strategy_ai_decisions：记录 AI 研判详情
        └── trade_signals：入选的生成 pending 信号
```

### 5.3 数据流

```
读：stock_basic + stock_indicators（行情技术面）
    market_breadth（市场宽度）
    trade_portfolio_snapshots/positions（当前持仓，供 AI 参考）

写：market_breadth（更新市场宽度）
    strategy_funnel（每只候选的筛选路径）
    strategy_ai_decisions（AI 对每只候选的研判）
    trade_signals（最终入选的买入信号）
    trade_holdings_review（AI 对现有持仓的审查建议）
```

# 第三部分：盘中交易

## 7. 盯盘系统概述

盯盘系统（Monitor）是整个系统中**唯一持续运行的实时模块**。它在交易日 09:25-15:00 期间每分钟扫描一次，负责判断市场状态、执行买卖决策、管理持仓风险。

代码组织：`Watcher` 主控制器（watcher.py）通过 7 个 Mixin 组装而成：

```
Watcher 主控制器（watcher.py）
    ├── MarketStateMixin      — 大盘状态判断（market_state.py, 2312 行）
    ├── BuyDecisionMixin      — 买入决策（buy_decision.py, 1600 行）
    ├── PositionRiskMixin     — 持仓风控（position_risk.py, 1174 行）
    ├── SectorContextMixin    — 板块趋势（sector_context.py, 750 行）
    ├── AbnormalMonitorMixin  — 异动检测（abnormal.py, 461 行）
    ├── ClosingDecisionMixin  — 尾盘决策（closing.py, 118 行）
    └── CloseSummaryMixin     — 收盘处理（close_summary.py, 183 行）
```

核心架构：`Watcher` 持有 `PaperAccount`（封装 `Portfolio`），所有 Mixin 通过 `self.paper_account` 操作模拟盘买卖。`RiskEngine` 负责风控检查，通过 `set_regime()` 注入 `MarketRegime` 感知大盘模式。

盯盘系统与外界的三条连接：
- **QMT HTTP**（192.168.1.33:5000）：获取个股实时价格
- **QMT Collector socket**（127.0.0.1:15555）：接收实时指数+全市场数据推送
- **Telegram Bot API**：推送消息 + 接收用户回复

以下第 8-17 节详细介绍盯盘系统的每个环节。

---

## 8. 数据从哪来

### 2.1 双通道数据源

盯盘系统从两个通道获取数据：

| 通道 | 连接方式 | 获取内容 | 频率 |
|------|----------|----------|------|
| QMT HTTP (192.168.1.33:5000) | HTTP 请求/响应 | 个股实时行情（价格、盘口、逐笔）、K线、下单 | 每轮 on-demand |
| QMT Collector (socket) | 长连接 TCP socket | 上证指数实时行情、全市场涨跌幅快照 | 实时推送（约每秒） |

**Collector 降级策略：** 当 Collector socket 未连接时，系统自动降级为从 DB `market_snapshots` 和 `sector_snapshots` 表读取最近的历史数据计算板块趋势，不影响核心盯盘逻辑。

### 2.2 QMT HTTP 接口

Windows 上的 QMT 开了一个 HTTP 服务（端口 5000），系统通过网络请求获取数据。以下是所有接口：

#### 行情类接口

| 接口 | 用途 | 调用时机 |
|------|------|---------|
| `GET /quote/000001.SH` | 获取单只股票实时行情 | 获取上证指数时 |
| `GET /quotes?codes=a,b,c` | 批量获取多只股票行情 | 每轮扫描，获取所有关注股票的价格 |
| `GET /all_quotes` | 获取全市场约 5000 只股票的涨跌幅 | 每 3 轮一次（约 3 分钟） |
| `GET /history/000001.SH?period=1m&count=240` | 获取分钟 K 线（开盘到现在的 1 分钟 K 线） | 启动恢复时、计算日内技术指标时 |
| `GET /tick/{code}` | 获取逐笔成交明细 | 评估买入候选时，分析大单流向 |

**单只股票行情返回的字段：**
- `lastPrice` — 最新价
- `preClose` — 昨收价
- `changePct` — 涨跌幅（%）
- `high` / `low` / `open` — 最高/最低/开盘价
- `amount` — 成交额
- `askVol` — 五档卖盘挂单量（数组，5 个值）
- `bidVol` — 五档买盘挂单量（数组，5 个值）

**全市场快照返回的字段（精简版）：**
- `lastPrice` — 最新价
- `changePct` — 涨跌幅
- `amount` — 成交额（用于计算板块资金量能）

#### 交易类接口

| 接口 | 用途 |
|------|------|
| `POST /order` | 下单（模拟盘） |
| `GET /instrument/{code}` | 获取合约信息（涨跌停价、总股本、流通股本） |

#### 核心表及其用途

**trade_signals（交易信号表）** — 这是最核心的表，所有买入决策都从这里出发：

| 字段 | 含义 | 示例 |
|------|------|------|
| stock_code | 股票代码 | 002371 |
| stock_name | 股票名称 | 北方华创 |

| 字段 | 含义 |
|------|------|
| signal_id | 关联的信号 ID |

| 字段 | 含义 |

## 9. 启动时做了什么

当你执行 `python main.py monitor` 时，系统按以下顺序初始化：

### Step 1：初始化连接

```
- 连接 SQLite 数据库
- 连接 Telegram Bot（用于发消息）
- 连接 QMT 客户端（用于获取个股实时行情）
- 连接 QMT Collector（socket 接收指数+全市场数据推送）
- 创建模拟盘账户 PaperAccount（初始资金 20 万）
- 创建风控引擎 RiskEngine
```

### Step 2：恢复昨日持仓

`PaperAccount.restore()` 从 `trade_portfolio_snapshots` 读取最新日终快照恢复账户现金和峰值资产，再从 `trade_portfolio_positions` 读取持仓明细恢复到内存。无需重放全部历史订单。恢复内容包括：
- 持仓股数、成本价（从 `trade_portfolio_positions` 表读取）
- 止损价、止盈价、移动止盈参数（从 `trade_signals` 表读取，存入 `_pos_meta`）
- 历史最高资产峰值（用于计算最大回撤）

### Step 3：清理旧快照

删除 3 天前的 `market_snapshots` 和 `sector_snapshots` 数据，防止数据库无限膨胀。

```sql
DELETE FROM market_snapshots WHERE trade_date < '2026-05-26'
DELETE FROM sector_snapshots WHERE trade_date < '2026-05-26'
```

### Step 4：恢复板块趋势

先从 `sector_snapshots` 表读取今天的板块历史数据。
如果这张表是空的（比如上午崩了下午重启），则从 `market_snapshots` 原始数据重建：
1. 读出今天所有全市场快照
2. 按时间分组
3. 每组按行业分类计算平均涨跌幅
4. 重建板块趋势时间序列

### Step 5：恢复指数全天走势

从 QMT 获取上证指数今天的分钟 K 线（最多 240 根），恢复到内存中：
- `_index_prices` — 每分钟的收盘价序列
- `_index_high` — 日内最高点
- `_index_low` — 日内最低点
- `_market_turnovers` — 每分钟成交额序列

这样即使下午才启动，也能知道上午发生了什么。

### Step 6：等待开盘

如果还没到 9:25（集合竞价结束），就每秒检查一次，等到开盘。

### Step 7：进入扫描循环

开始每分钟一轮的主循环。

## 10. 盘中每分钟在干什么

### 4.1 一天的时间表

| 时间 | 阶段 | 做什么 |
|------|------|--------|
| 00:00–09:25 | 盘前等待 | 每 30 秒检查一次是否到开盘时间 |
| 09:25 | 集合竞价结束 | 第一轮扫描，推送开盘决策汇总 |
| 09:30–11:30 | 上午连续竞价 | 正常扫描循环（每 60 秒一轮） |
| 11:30–13:00 | 午休 | 每 30 秒检查一次是否到下午开盘 |
| 13:00–14:30 | 下午交易 | 正常扫描循环 |
| 14:30 | 尾盘 | 触发一次尾盘决策 |
| 14:30–15:00 | 尾盘交易 | 继续扫描但不重复触发尾盘决策 |
| 15:00 | 收盘 | 保存快照、过期未触发信号、推送收盘持仓报告（模拟盘→群聊，实盘→私聊）、退出进程 |

### 4.2 每轮扫描的完整流程

每一轮（约 60 秒）执行以下步骤：

```
第 0 步：接收 Collector 实时数据
    — 从 DataCollectorClient socket 读取指数+全市场行情推送
    — 去重：跳过 DB 已恢复的旧数据
    — 更新 _index_prices / _index_high / _index_low

第 1 步：拉取用户 Telegram 回复
    — 检查是否有人回复了"成交 XXX"或"再等 N 分钟"等指令

第 2 步：确定监控范围
    — 持仓中的股票 + 有 pending 信号的股票 + 复盘推荐的股票
    — 通常 20-40 只

第 3 步：批量获取实时价格
    — 调用 QMT /quotes 接口，一次获取所有监控股票的价格
    — 同时缓存涨跌停价格

第 4 步：更新持仓市值
    — 把最新价格写入 paper_account，更新每只持仓的浮动盈亏

第 5 步：检查最大回撤
    — 总账户从最高点回撤超过 15%？触发警报，阻止新买入

第 6 步（每 3 轮一次）：更新板块趋势
    — Collector 未连接时从 DB 读取全市场快照计算板块均值
    — Collector 连接时数据已实时推送，此步跳过
    — 追踪连续性（连续多少轮同方向）
    — 计算板块相对大盘的强度

第 7 步：第一层扫描 — 大盘状态检测
    — 调用 _check_market_state() 返回 MarketRegime 对象
    — regime.allow_buy → 是否允许买入
    — regime.position_mult → 仓位乘数
    — regime.stop_mult → 止损宽度乘数
    — 更新 risk_engine 市场环境打分

第 8 步：指数技术指标拐点检测
    — MACD/RSI/KDJ 拐点，状态变化时推送

第 9 步：持仓止损止盈检查（对所有持仓）
    — 读大盘 risk_level + 板块趋势动态调整触发线
    — 止损/止盈/移动止盈/利润回撤/时间止损
    — T+1 保护：当日买入的不触发
    — 触发后 paper_account.sell() 执行卖出

第 10 步：AI 信号检查 → 送入统一买入候选管线

第 11 步：已买入信号盯盘（更新最高价、浮盈追踪）

第 12 步：复盘推荐检查 → 送入统一买入候选管线

第 13 步：止损提醒循环

第 14 步（如果是第一轮）：推送开盘决策汇总

第 15 步（每 50 轮一次）：板块热度排名

第 16 步（每 3 轮一次）：异动检测

第 17 步（每 15 轮一次）：主动换仓评估

第 18 步（14:30 后一次）：尾盘决策
```

## 11. 怎么判断大盘好不好

大盘判断在 `MarketStateMixin`（market_state.py）中，分为多个层次。

### 5.1 获取上证指数

专门调用 `/quote/000001.SH` 获取上证指数的：
- 最新价
- 昨收价
- 涨跌幅
- 成交额

每轮扫描都更新一次。价格序列追加到 `_index_prices` 列表中，全天保留。

### 5.2 市场模式分类

基于上证指数全天的价格轨迹，分成 5 种模式。**不再使用固定三段切分**，改为多时间窗口滚动对比 + 日内 EMA：

- **短期窗口（15 点 ≈ 15 分钟）**：捕捉最近动量
- **中期窗口（60 点 ≈ 60 分钟）**：判断盘中趋势
- **日内 EMA12/EMA26**：动态均线参考（替代固定的三段均值比较）

#### 模式 1：normal（正常）
- 条件：数据点不足 20 个，或不满足以下任何特殊模式
- 含义：大盘正常波动
- 买入：✅ 允许
- 消息：无

#### 模式 2：panic（恐慌）
- 条件（必须同时满足）：
  1. 日内振幅 > 1.5%
  2. 当前价格在日内低位附近（距离日内低点不到振幅的 10%）
  3. 短期跌幅 > 中期跌幅 × 1.0（加速阈值从 1.5× 降至 1.0×，更早捕捉恐慌）
  4. 短期跌幅 > 0.5%
- 含义：市场在加速下跌，恐慌情绪蔓延
- 买入：🚫 禁止
- 消息：`🚨 恐慌下跌: 上证 XXXX.XX 加速下探，日内振幅X.X%，建议暂停所有买入，考虑减仓`

#### 模式 3：one_sided（单边下跌）
- 条件（必须同时满足）：
  1. 价格持续在日内 EMA12 下方运行
  2. 短期窗口均价继续下降
  3. 中期重心下移（当前中期均价 < 前段中期均价，跌幅 > 0.5%）
- 含义：全天持续阴跌，重心不断下移
- 买入：🚫 禁止
- 消息：`⚠️ 单边下跌: 上证 XXXX.XX，重心持续下移，暂停买入`

#### 模式 4：dead_cat（死猫跳/弱势反弹）
- 条件（必须同时满足）：
  1. 短期反弹（短期变化 > +0.2%）
  2. 价格已从低点回升（> 0.3%）
  3. 但当前价格没有回到日内 50% 分位以上
  4. 价格 ≤ 日内 EMA12（未站上动态均线）
- 含义：有反弹但力度很弱，可能是诱多
- 买入：🚫 禁止
- 消息：`⚠️ 弱势反弹(疑似死猫跳): 上证 XXXX.XX，反弹未过50%分位，暂不跟进`

#### 模式 5：v_reversal（V 型反转）
- 条件（必须同时满足）：
  1. 短期反弹（短期变化 > +0.2%）
  2. 价格从低点回升 > 0.3%
  3. 当前价格回到了日内 50% 分位以上
  4. 价格站上日内 EMA12
  5. **技术确认**（至少满足一项）：MACD 金叉、RSI 从超卖回升、KDJ 金叉
- 含义：深跌后强势回升，有技术指标确认
- 买入：✅ 允许
- 消息：`🔄 V型反转迹象: 上证 XXXX.XX 深跌后回升至50%分位以上，恢复买入信号`

### 5.3 市场环境判定（多维度牛熊打分）

风控引擎中的市场环境不再仅靠 MA20 ± 3% 简单判断，而是用 **6 维度综合打分**：

| 维度 | 数据来源 | 权重 | 正面条件 | 负面条件 |
|------|----------|------|----------|----------|
| 价格 vs MA20 | DB stock_basic | ±3 | 偏离 > +5% → +3 | 偏离 < -5% → -3 |
| 价格 vs MA60 | DB stock_basic | ±2 | 偏离 > +3% → +2 | 偏离 < -3% → -2 |
| 量能趋势 | 全市场近5天成交额 | ±2 | 放量 > +15% → +2 | 缩量 > -15% → -2 |
| 涨跌家数比 | 全市场快照 | ±2 | 比值 > 2.0 → +2 | 比值 < 0.5 → -2 |
| 日内振幅 | 上证 high/low | -1 | — | 振幅 > 3% → -1 |
| 活跃板块数 | 板块统计 | ±1 | ≥ 5 个 → +1 | ≤ 1 个 → -1 |

**总分 → 环境判定：**
- `≥ 3` → **bull**（牛市），仓位上限 80%
- `0 ~ 2` → **swing**（震荡），仓位上限 50%
- `< 0` → **bear**（熊市），仓位上限 20%

### 5.4 传统阈值补充

即使模式为 normal，还会检查：

| 条件 | 后果 | 消息 |
|------|------|------|
| 上证跌幅 > 2%（`INDEX_HALT_PCT = -0.02`） | 🚫 熔断，禁止所有买入 | `🚨 大盘熔断: 上证跌幅 -X.X%，暂停所有买入信号` |
| 上证跌破 MA20 且跌幅 > 1%（`INDEX_DANGER_PCT = -0.01`） | 🚫 暂停买入 | `⚠️ 大盘偏弱: 上证 XXXX.XX 跌破 MA20(XXXX.XX)，跌幅 -X.X%，暂停买入` |

### 5.5 结构性单边下跌判断

即使模式分类没触发，还有一个更严格的结构性判断（`_is_index_downtrend`）：

三个条件**必须同时满足**：
1. **价格在日内下 1/3**：当前价 < 日内低点 + 振幅/3
2. **重心下移**：最近 10 轮均价 < 前 10 轮均价
3. **全市场普跌**：跌家数 > 2 × 涨家数（或涨家数为 0）

满足 → 🚫 暂停买入

### 5.6 量价背离检测

每轮都会检查最近 12 个数据点的量和价是否背离：

**怎么算的：**
1. 取最近 12 轮的上证价格和成交额
2. 把每日累计成交额转成每轮增量
3. 分成前后两半，比较增量均值的变化

**两种背离：**

| 类型 | 条件 | 含义 | 消息 |
|------|------|------|------|
| 诱多 | 价格涨 > 0.3% 但量缩 > 15% | 上涨没有量能支撑，可能是假突破 | `⚠️ 量价背离(诱多): 上证近X分钟涨X.X%，但成交额缩X% → 上涨缺乏量能支撑，注意诱多风险` |
| 恐慌放量 | 价格跌 > 0.3% 但量增 > 15% | 恐慌盘在涌出，加速赶底 | `⚠️ 量价背离(恐慌): 上证近X分钟跌X.X%，但成交额放X% → 恐慌盘涌出，关注是否加速赶底` |

### 5.7 指数技术指标拐点

基于上证指数的分钟级 K 线（每 5 个扫描点 ≈ 1 分钟），计算 MACD、RSI、KDJ，检测以下拐点：

**MACD 交叉：**
- 金叉（DIF 上穿 DEA）→ 多头信号
- 死叉（DIF 下穿 DEA）→ 空头信号

**RSI 极值：**
- RSI6 < 20 → 超卖
- RSI6 > 80 → 超买
- RSI12 < 20 → 超卖
- RSI12 > 80 → 超买

**KDJ 极值：**
- J < 0 → 超卖
- J > 100 → 超买

**背离：**
- 底背离：价格新低但 MACD DIF 没有新低 → 下跌动能衰竭
- 顶背离：价格新高但 MACD DIF 没有新高 → 上涨动能衰竭

**只在状态变化时推送一次**，不是每轮都发。

### 5.8 组合信号的操作建议

系统会根据信号组合自动生成操作建议，优先级从高到低：

| 信号组合 | 建议 |
|----------|------|
| 底背离 | "下跌动能衰竭，关注反转确认后可小仓位试探，建议等MACD金叉或RSI回升配合" |
| 顶背离 | "上涨动能衰竭，建议减仓或观望，不宜追高，持仓注意止盈" |
| 超卖 + MACD 金叉 | "超卖+金叉共振，技术面转多信号较强，可考虑分批低吸，注意止损" |
| 超卖 + MACD 死叉 | "超卖但MACD尚未转多，恐慌赶底阶段，等金叉确认后再入场更安全" |
| 超买 + MACD 死叉 | "超买+死叉共振，技术面转空信号较强，建议减仓避险，至少不加仓" |
| 超买 + MACD 金叉 | "强势但已过热，持仓可继续持有但不宜加仓，关注RSI回落或死叉出场信号" |
| 仅超卖 | "超卖区，关注企稳信号（缩量止跌/下影线），不宜追空，可准备低吸" |
| 仅超买 | "超买区，追高风险大，建议等回调或至少等RSI回到70以下再考虑" |

### 5.9 AI 大盘波动分析

当上证近 3 轮波动 ≥ 0.5% 时，会把当前的分钟级 MACD/RSI/KDJ/均线数据发给 AI（DeepSeek 或 Qwen），让它分析：
1. 这波急跌/急拉会继续还是反转？
2. 到什么点位可能企稳？
3. 当前应该追/等/减/守？

AI 的回复会附在大盘波动消息后面。如果 AI 不可用，会用内置的简单建议替代。

### 5.10 双重回撤保护

系统有两层回撤保护（`_check_max_drawdown`）：

**第一层：日内熔断**
- 当日亏损超过总资产的 3% → 触发警报
- 消息：`🚨 日内熔断: 日亏损 X.X% — 建议暂停所有买入，评估是否减仓`
- 阻止所有新买入

**第二层：总账户回撤**
- 总账户从历史最高点回撤 ≥ 15%
- 触发一次警报：`🚨 最大回撤警报 — 建议立即清仓所有持仓`
- `drawdown_halt = True`：阻止所有新买入
- 不影响止损卖出（该止损照样执行）

### 5.11 市场情景预测引擎（前瞻预判）

在模式分类之前，先运行一个**贝叶斯概率引擎**，预测 8 种"正在形成"的市场情景。这是**前瞻性预判**而非事后模式识别。

#### 8 种预测情景

| 情景 | 方向 | 含义 | 预设行动 |
|------|------|------|----------|
| normal_stable | neutral | 正常平稳，无异常信号 | 正常操作 |
| developing_uptrend | bullish | 正在形成的上涨趋势 | 可适度积极 |
| developing_downtrend | bearish | 正在形成的下跌趋势 | 收紧止损，减少买入 |
| accelerating_down | bearish | 下跌加速 | 暂停买入，考虑减仓 |
| accelerating_up | bullish | 加速上涨（警惕冲顶） | 不追高，收紧止盈 |
| potential_reversal_up | bullish | 潜在向上反转 | 准备抄底，等确认 |
| potential_reversal_down | bearish | 潜在向下反转 | 获利了结，收紧止损 |
| dead_bounce | bearish | 弱势反弹/死猫跳 | 不跟进，趁反弹减仓 |

#### 工作原理（三层架构）

**第 1 层：MicroSignals** — 每轮扫描从价格序列中提取 7 个微观信号（price_velocity / price_accel / ema12_pos / vol_pulse / breadth_pct / rsi_signal / macd_signal）。

**第 2 层：SCENARIO_SIGNALS** — 每个情景有 `confirm` lambda 条件（满足 → +0.15）和 `reject` lambda 条件（满足 → -0.25），不是硬性判定而是概率调制。

**第 3 层：Bayesian 概率更新** — `raw[name] = prev × clamp(signal_adj, 0.5, 1.5)`，未确认情景 ×0.92 时间衰减，最后归一化到总和 1.0。

#### 输出：MarketOutlook

- `primary` — 概率最高的主情景（含 confirm_at/invalidate_at 关键关卡）
- `alternatives` — 次高概率的备选情景
- `key_support/key_resistance` — 关键支撑/阻力位
- `bias` — 市场方向（bullish/bearish/neutral）
- `urgency` — 紧急程度

#### 紧急程度阈值

| 主情景概率 | 紧急程度 | 含义 |
|-----------|----------|------|
| ≥ 70% | critical | 立即执行行动 |
| ≥ 55% | act | 提前调整策略 |
| ≥ 35% | watch | 关注变化 |
| < 35% | none | 无特殊行动 |

#### 与 ASSESS 阶段的融合

outlook 传入 `_assess_regime()`，在模式判定基础上叠加前瞻调整：
- 偏空 + urgency ≥ act → risk_level 升级一档
- 偏空 + urgency ≥ critical → 强制 allow_buy=False
- 偏多 + urgency ≥ act → position_mult × 1.2
- 偏多 + urgency ≥ watch → stop_mult × 0.9

#### 告警推送

在主情景切换 / 概率大幅变化(>20%) / 紧急程度升级 / 关键关卡接近(<0.3%) 时推送 🔮 市场预判消息。

## 12. 怎么判断板块强不强

板块趋势追踪在 `SectorContextMixin`（sector_context.py）中。同时追踪**行业板块**和**概念板块**两类。

### 6.1 数据从哪来

每 3 轮（约 3 分钟）：
1. QMT `/all_quotes` → 全市场 ~5000 只股票的涨跌幅
2. 按行业分类（行业信息来自 `stock_basic` 表的 `industry` 字段）
3. 同时按概念板块分类（概念信息来自 `stock_basic` 表的 `concept` 字段）
4. 每个行业/概念板块计算平均涨跌幅
5. 追加到该板块的历史序列中

**板块资金量能：** 在聚合板块数据的同时，累加每只股票的 `amount`（成交额）字段，计算板块总成交额。通过比较当前轮和上轮板块成交额，得出 `vol_ratio`（量比），用于判断资金是否在流入该板块。

### 6.2 概念板块趋势

概念板块趋势判定与行业板块逻辑相同（线性回归斜率 + 累计变化），但判断阈值不同：

| 板块涨跌幅 | 概念板块评分 |
|-----------|-------------|
| > +1.0% | +1 分（强势） |
| < -1.0% | -1 分（弱势） |

取股票所属的前 5 个概念板块，综合评分：
- `≤ -2`：多数概念板块走弱 → 在买入决策中直接拒绝
- `< 0`：概念板块偏弱 → 仓位打 6 折
- 其他：不影响

### 6.3 行业趋势方向怎么判定

板块趋势判定函数：`_get_sector_trend(code)`

**输入：** 股票代码
**输出：** 类似 `板块半导体 持续走弱 加速 -6.6% 弱于大盘 普跌` 的字符串

**判断步骤：**

**第 1 步：取该股票所属行业的历史数据**
- 从 `_sector_trend_history[industry]` 取今天所有快照点的均值序列
- 如果该行业数据不足 2 个点，直接返回简单格式

**第 2 步：计算全天累计变化**
```
累计变化 = 最新均值 - 最早均值
```
比如半导体最早是 +0.5%，最新是 -6.1%，累计 = -6.6%

**第 3 步：计算线性回归斜率**

对所有快照点做最小二乘法拟合一条直线，斜率反映全天的趋势方向。公式：
```
x_mean = (n-1) / 2
y_mean = sum(所有均值) / n
斜率 = sum((i - x_mean) × (均值[i] - y_mean)) / sum((i - x_mean)²)
```

用斜率而不是简单的"首尾差"来判断方向，是因为斜率反映了**全天所有数据点的整体趋势**，不会被个别异常点干扰。

**第 4 步：判定趋势方向**

| 条件 | 判定结果 |
|------|----------|
| 斜率 > +0.003 且 数据点 ≥ 5 且 累计 > +0.3% | **持续走强** |
| 斜率 > +0.003 且 数据点 ≥ 5 但 累计 ≤ +0.3% | 走强 |
| 斜率 < -0.003 且 数据点 ≥ 5 且 累计 < -0.3% | **持续走弱** |
| 斜率 < -0.003 且 数据点 ≥ 5 但 累计 ≥ -0.3% | 走弱 |
| 累计绝对值 < 0.3% | 横盘 |
| 累计 > 0（其他情况） | 走强 |
| 累计 < 0（其他情况） | 走弱 |

**关键设计**：
- `n >= 5` 要求至少有 5 个快照点（约 15 分钟数据），防止开盘几分钟的随机波动被误判为"持续"
- 斜率阈值 `±0.003` 很小，因为板块均值每轮变化通常只有千分之几
- "持续" = 斜率方向 + 累计方向一致 + 够多数据点 → 全天趋势明确

### 6.4 加速度：趋势在加剧还是减缓

加速度 = 最近半段的斜率 vs 全段斜率：

| 趋势方向 | 最近半段斜率 vs 全段斜率 | 加速度 |
|----------|------------------------|--------|
| 走强中 | 最近半段 > 全段 × 1.5 | 加速（涨得越来越快） |
| 走弱中 | 最近半段 < 全段 × 1.5 | 加速（跌得越来越快） |
| 走强中 | 最近半段 < 全段 × 0.3 | 趋缓（涨势在减弱） |
| 走弱中 | 最近半段 > 全段 × 0.3 | 趋缓（跌势在减弱） |

### 6.5 相对强度：板块 vs 大盘

```
相对强度 = 板块均值 - 全市场均值
```

| 相对强度 | 标签 |
|----------|------|
| > +0.5% | 强于大盘 |
| -0.5% ~ +0.5% | （不显示） |
| < -0.5% | 弱于大盘 |

### 6.6 板块广度：普涨还是普跌

```
板块广度 = (涨家数 - 跌家数) / 总家数
```

| 广度 | 标签 |
|------|------|
| > 0.4 | 普涨（多数股票在涨） |
| -0.4 ~ 0.4 | （不显示） |
| < -0.4 | 普跌（多数股票在跌） |

### 6.7 板块趋势在买入决策中的使用

板块趋势贯穿整个买入决策，有四个层级的影响：

| 板块趋势 | 区内买入 | 回调买入 | 仓位修正 | 高于买入区 |
|----------|----------|----------|----------|-----------|
| **持续走弱** | 🚫 直接拒绝 | 🚫 直接 watching | ×0.3 | 跳过 |
| 走弱 | ⚠️ 仓位 ×0.5 | 评分 -1 | ×0.6 | 跳过 |
| 横盘 | 不影响 | 不影响 | 不影响 | 跳过 |
| 走强 | 不影响 | 评分 +1 | ×1.2 | 跳过 |
| **持续走强** | ✅ 仓位 ×1.2 | 评分 +3 | ×1.3 | 📈 推送追高提醒 |

这是最重要的设计原则：**板块持续走弱时，坚决不逆势买入。板块持续走强时，顺势加仓。**

### 6.8 开盘决策汇总

每天第一轮扫描时（9:25 后），推送一条汇总消息，包含：

1. **上证指数**：点位、涨跌幅、相对 MA20 位置、大盘状态
2. **当前持仓**：每只的盈亏、是否 T+1 锁定、是否触发止损/止盈
3. **买入区信号**：价格已在买入区内的信号（标为 🔴）
4. **待观察信号**：价格还未进入买入区（标为 👀，标注"高于"或"低于"买入区）
5. **板块集中度提醒**：如果某个板块有 ≥ 3 只信号，特别提醒注意分散风险

## 13. 怎么决定买不买

买入决策在 `BuyDecisionMixin`（buy_decision.py）中。这是系统最复杂的部分。

### 7.1 两个信号来源

系统从两个渠道获取买入候选，汇入同一条处理管线：

**渠道 A：AI 信号（`_check_signals`）**
- 来源：`trade_signals` 表，`status = 'pending'` 且 `account = 'paper'`
- 由盘前的 AI 分析流程生成，包含精确的买入区间、止损价、止盈价

**渠道 B：复盘推荐（`_check_review_picks`）**
- 来源：`stock_tracker` 表中的复盘推荐记录
- 如果该推荐已经入库为 trade_signal，则由渠道 A 处理，渠道 B 跳过
- 买入区间由 `ReviewPickMonitor` 动态提供

### 7.2 统一候选处理管线

所有候选进入 `_check_buy_candidates()`，按当前价格分三种情况：

```
当前价格 vs 买入区间 [buy_min, buy_max]

情况 1：price > buy_max（高于买入区）
    ├── 板块持续走强？→ 📈 推送追高提醒，但不自动买
    └── 否则 → 跳过，不处理

情况 2：price >= buy_min AND price <= buy_max（在买入区内）
    ├── 先检查防重复推送
    ├── 大盘不好？→ ⏸️ 推送暂停消息
    ├── 涨停了？→ 🚫 推送涨停消息
    ├── 都没问题 → 做 9 因子评估
    │   ├── 有 reject → ⛔ 推送跳过原因，不买
    │   ├── 有 warn → ⚠️ 减仓执行
    │   └── 通过 → 🔴 推送买入信号 + 模拟盘执行
    └── 模拟盘满仓（5只）→ 推送信号但不执行

情况 3：price < buy_min（低于买入区）
    └── 做 8 因子评分
        ├── abandon（评分 < -4）→ ❌ 放弃信号，标记 expired
        ├── watching（评分 -4~2）→ 👀 不买不弃，继续观察
        └── opportunity（评分 ≥ 3）→ 🟢 回调买入，模拟盘执行
```

### 7.3 区内买入的 9 个因子（`_evaluate_buy_decision`）

返回 `(是否允许, 原因, 仓位倍数)`。

仓位倍数从 1.0 开始，每经过一个因子都可能乘以一个系数。最终 ≥ 0.5 才执行。

#### 因子 1：行业板块趋势

- 数据来源：`_get_sector_trend(code)`
- 判断依据：板块全天线性回归斜率 + 累计变化

| 板块状态 | 效果 | 仓位影响 |
|----------|------|----------|
| 持续走弱 | 🚫 拒绝买入 | size_mul = 0（强制归零） |
| 走弱 | ⚠️ 警告 | ×0.5 |
| 持续走强 | ✅ 加分 | ×1.2 |
| 走强/横盘 | 不影响 | — |

#### 因子 1b：概念板块趋势

- 数据来源：`_get_concept_trend_score(code)`，取前 5 个概念板块
- 判断依据：各概念板块实时涨跌幅，+1%以上算强、-1%以下算弱

| 概念板块评分 | 效果 | 仓位影响 |
|-------------|------|----------|
| ≤ -2（多数走弱） | 🚫 拒绝买入 | size_mul = 0 |
| < 0（偏弱） | ⚠️ 警告 | ×0.6 |

#### 因子 2：买入区位置

- 数据来源：当前价格 vs 买入区上下限
- 计算公式：`zone_pos = (price - buy_min) / (buy_max - buy_min)`

| zone_pos | 判定 | 效果 |
|----------|------|------|
| ≥ 85% | 买入区顶部 | 🚫 拒绝（不追高） |
| ≥ 65% | 买入区偏上 | ⚠️ 警告，×0.7 |
| ≤ 33% | 买入区下沿 | 在仓位计算中 ×1.1（激进） |

#### 因子 3：布林带位置

- 数据来源：`stock_indicators` 表的 `bb_pct_b`（%B 指标）
- %B = (价格 - 布林下轨) / (布林上轨 - 布林下轨) × 100
- 这是**昨日收盘**数据，反映中长期位置

| %B | 判定 | 效果 |
|----|------|------|
| ≥ 90 | 布林带超买 | 🚫 拒绝（回调风险高） |
| ≥ 75 | 布林带偏上 | ⚠️ 警告，×0.8 |
| ≤ 15 | 布林带超卖 | 不拒绝（下轨附近是好买点） |

**额外判断：回踩关键支撑**（价格在布林下轨或 MA20 的 ±3% 内且没有其他拒绝原因）→ 加分 ×1.2

#### 因子 4：均线关系

- 数据来源：`stock_indicators` 表的 `ma5`, `ma10`, `ma20`

| 条件 | 判定 | 效果 |
|------|------|------|
| 价格 < MA5 且 < MA10 且 < MA20，同时 MA5 < MA10 < MA20 | 空头排列 + 破位 | 🚫 拒绝（接飞刀） |
| 价格 < MA5 且 < MA10 且 < MA20，但均线不是标准空头排列 | 价格低于所有均线 | ⚠️ 警告，×0.7 |

#### 因子 5：日内分钟级技术指标

- 数据来源：QMT 分钟 K 线（最近 120 根），实时计算 RSI/MACD/KDJ
- 每轮扫描缓存一次（同一轮内多个候选共享）

**RSI：**

| RSI6 | 判定 | 效果 |
|------|------|------|
| ≥ 85 | 极度超买 | 🚫 拒绝（追高风险极大） |
| ≥ 75 | 超买 | ⚠️ 警告，×0.7 |
| ≤ 20 | 深跌超卖 | 加分 ×1.1（可能是好买点） |

**MACD：**

| 状态 | 判定 | 效果 |
|------|------|------|
| 空头 + bar < -0.5 | 强烈空头 | 🚫 拒绝（下跌动能未衰竭） |
| 空头 + bar < -0.1 | 偏空 | ⚠️ 警告，×0.8 |
| 多头 + bar > 0.2 | 偏多 | 加分 ×1.1 |

**KDJ：**

| J 值 | 判定 | 效果 |
|------|------|------|
| > 100 | 极度超买 | 🚫 拒绝 |
| > 85 | 超买 | ⚠️ 警告，×0.7 |
| < 0 | 深跌超卖 | 加分 ×1.1 |

**KDJ 死叉**（K < D 且 J < 50）→ ⚠️ 警告，×0.8

**价格 vs 日内 MA5：**

| 偏离 | 判定 | 效果 |
|------|------|------|
| < -3% | 远离日内均线 | 🚫 拒绝（短期急跌接飞刀） |
| < -1.5% | 低于日内均线 | ⚠️ 警告 |

#### 因子 6：五档盘口买卖力量

- 数据来源：QMT `/quote` 返回的 `bidVol`（买盘）和 `askVol`（卖盘）
- 计算：取前 5 档总量，`买盘占比 = 买盘总量 / (买盘总量 + 卖盘总量)`

| 买盘占比 | 判定 | 效果 |
|----------|------|------|
| ≤ 30% | 卖盘沉重 | 🚫 拒绝 |
| ≤ 42% | 卖压偏大 | ⚠️ 警告，×0.85 |
| ≥ 70% | 买盘强劲 | 加分 ×1.1 |

#### 因子 7：大单流向

- 数据来源：QMT `/tick` 返回的逐笔成交
- 方法：统计最近 200 笔成交中，单笔 ≥ 5 万元的大单的买卖方向

| 大单买入占比 | 判定 | 效果 |
|-------------|------|------|
| ≤ 35% | 大单卖出主导 | 🚫 拒绝 |
| ≤ 45% | 大单偏卖 | ⚠️ 警告，×0.8 |
| ≥ 65% | 大单买入主导 | 加分 ×1.1 |

#### 因子 8：涨跌停空间

- 数据来源：QMT `/instrument` 返回的 `upStopPrice`（涨停价）、`downStopPrice`（跌停价）

| 条件 | 判定 | 效果 |
|------|------|------|
| 距涨停 < 2% | 追板风险极高 | 🚫 拒绝 |
| 距涨停 < 4% | 上行空间有限 | ⚠️ 警告，×0.8 |
| 距跌停 > 15% | 下方风险空间大 | 🚫 拒绝 |

#### 因子 9：昨日趋势背景 + 今日实时因子

- 数据来源：`stock_basic` 表（昨日主力资金、MA5 角度）+ QMT 实时（今日日内位置）+ 5 分钟 K 线 MACD

**昨日因子（来自 `_get_context_factors`）：**

| 因子 | 来源 | 正面 | 负面 |
|------|------|------|------|
| 主力净流入占比 | stock_basic.main_force_ratio | > 5% → ×1.1 | < -5% → 🚫 拒绝；-5%~-2% → ⚠️ |
| MA5 角度 | stock_basic.ma5_angle | — | < -2 → 🚫 拒绝（加速下行）；-2~0 → ⚠️ ×0.85 |
| 日线 MACD | stock_indicators | DIF > DEA + bar > 0.2 → ×1.05 | DIF < DEA + bar < -0.3 → ⚠️ ×0.85 |
| 日线 KDJ J | stock_indicators | J < 0 → ×1.1 | J > 100 → 🚫 拒绝；J > 85 → ⚠️ ×0.8 |
| BBI 多空线 | stock_indicators | 价格 > BBI | 价格 < BBI×0.95 → ⚠️ ×0.85 |

**今日实时因子：**

| 因子 | 来源 | 正面 | 负面 |
|------|------|------|------|
| 日内位置 | QMT high/low | day_pos < 0.15 + MA5 向上 → ×1.1 | day_pos > 0.9 → ⚠️ ×0.85 |
| 5分钟 MACD | QMT 5min K线 | 多头 + bar > 0.1 → ×1.05 | 空头 + bar < -0.2 → ⚠️ ×0.85 |
| 布林带宽 | stock_indicators.bb_width | — | > 40 → ⚠️ ×0.8（波动剧烈） |

### 7.4 大盘入场规则过滤（entry_rule）

在进入 9 因子评估之前，`_check_buy_candidates` 会先根据 MarketRegime 的 `entry_rule` 过滤：

| entry_rule | 含义 | 过滤条件 |
|-----------|------|----------|
| standard | 正常入场 | 不限制 |
| confirm | 需确认信号 | zone_pos > 0.5 → 暂缓，等回调到区间下半部 |
| pullback | 等回调买入 | zone_pos > 0.4 → 暂缓，不追高 |
| range_boundary | 宽幅震荡 | zone_pos > 0.25 → 等区间下沿再入场 |
| next_day | 次日再看 | 直接跳过，今日不追（尾盘拉升等场景） |
| none | 禁止入场 | 直接跳过（panic/one_sided 等模式） |

### 7.5 大盘止损宽度调整（stop_mult）

模拟盘买入时，`_execute_paper_buy` 会根据 MarketRegime 的 `stop_mult` 动态调整止损宽度：

```
effective_sl = price - (price - sl) × stop_mult
```

| stop_mult | 效果 | 典型场景 |
|-----------|------|----------|
| 1.0 | 不调整 | normal/uptrend |
| 0.7~0.8 | 止损收紧 20-30% | melt_up/v_reversal |
| 1.2 | 止损放宽 20% | one_sided/wide_choppy |
| 1.5 | 止损放宽 50% | panic/late_dump |

stop_mult > 1 表示大盘波动剧烈时放宽止损线以避免被噪音洗出；stop_mult < 1 表示大盘危险时收紧止损以更快离场。

### 7.6 回调买入的 8 个因子评分（`_evaluate_below_zone`）

当价格低于买入区下限时，不是简单放弃，而是做一套打分系统。返回 `(action, reason, size_mul)`。

**评分机制：** 每个因子加分或扣分，总分决定动作。

#### 因子 1：偏离买入区幅度

| 低于买入区的幅度 | 评分 |
|-----------------|------|
| ≤ 2%（微幅偏离，接近买入区） | +2 |
| 2%~4%（中等偏离） | 0 |
| 4%~7%（较大偏离） | -2 |
| > 7%（大幅偏离，买入区可能已失效） | -5 |

#### 因子 2：距离关键支撑

检查价格是否在以下支撑位附近（±2-3%）：

| 价格位置 | 评分 |
|----------|------|
| 触及布林下轨（±2%） | +4 |
| 触及 MA20（±2%） | +4 |
| 触及 MA60（±3%） | +4 |
| 在 MA20 上方 | +1 |
| 跌破 MA60 | -3 |

#### 因子 3：日内技术指标

| 指标 | 加分条件 | 扣分条件 |
|------|----------|----------|
| RSI6 | ≤ 25 → +3（超卖）；≤ 35 → +1 | ≥ 70 → -1 |
| KDJ J | < 0 → +2 | — |
| KDJ 交叉 | K > D 且 J < 20 → +2（低位金叉） | — |
| MACD 方向 | 多头 → +1 | 空头 + bar < -0.3 → -2 |
| 价格 vs MA5 | — | < -5% → -3（急跌）；< -2% → -1 |

#### 因子 4：大单方向

| 大单情况 | 评分 |
|----------|------|
| 大单买入占比 ≥ 60% | +3（有资金承接） |
| 大单卖出占比 ≥ 60% | -3（主力出货） |

#### 因子 5：盘口支撑

| 盘口情况 | 评分 |
|----------|------|
| 买盘占比 ≥ 65% | +2（买盘厚实，有支撑） |
| 卖盘占比 ≥ 65% | -2（卖盘压力大） |

#### 因子 6：行业板块趋势

| 板块状态 | 评分 |
|----------|------|
| 持续走弱 | **直接返回 watching**，不追回调 |
| 持续走强 | +3 |
| 走强 | +1 |
| 走弱 | -1 |

#### 因子 6b：概念板块趋势

- 取前 5 个概念板块，按实时涨跌幅评分
- `≤ -2`：**直接返回 watching**，不追回调
- 其他：评分叠加（-3 ~ +3）

#### 因子 7：成交量验证

比较最近一半 tick 和之前一半 tick 的成交量变化：

| 量能变化 | 评分 | 含义 |
|----------|------|------|
| vol_ratio < 0.5 | +2 | 缩量下跌 = 正常回调 |
| vol_ratio > 2 | -2 | 放量下跌 = 恐慌抛售 |

#### 因子 8：昨日趋势背景 + 今日实时

| 因子 | 加分 | 扣分 |
|------|------|------|
| 昨日主力净流入占比 | > 3% → +3（洗盘） | < -3% → -3（真破位） |
| MA5 角度 | > 1 → +1（仍向上） | < -3 → -3（加速下行） |
| 日内位置 | day_pos < 0.1 → +1 | — |
| 日线 MACD | DIF > DEA → +1 | DIF < DEA + bar < -0.5 → -2 |
| 日线 KDJ J | < 0 → +2 | > 90 → -1 |
| BBI | — | 价格 < BBI×0.9 → -2 |
| 5分钟 MACD | 多头 → +1 | 空头 → -1 |

#### 总分 → 动作映射

| 总分 | 动作 | 仓位 | 含义 |
|------|------|------|------|
| ≥ 6 | opportunity | min(1.0, 0.5+得分×0.05) | 回调至支撑区，择机买入 |
| 3~5 | opportunity | min(0.7, 0.5+得分×0.05) | 止跌迹象，小仓位试探 |
| 0~2 | watching | None | 未企稳，继续观察 |
| -4~-1 | watching | None | 偏弱，等更明确信号 |
| < -4 | abandon | None | 破位下行，买入区已失效 |

### 7.7 预测性买入区接近检测

当价格高于买入区但距上限 ≤ 3% 时，不是简单跳过，而是结合情景引擎做**前瞻性预告**：

| 条件 | 动作 |
|------|------|
| 距买入区 ≤ 3% + 市场偏空 + urgency ≥ act | 📉 "买入区接近"告警，预告价格可能很快进入买入区 |
| 其他（高于买入区 > 3% 或市场正常） | 照常跳过或触发追高提醒 |

目的是让用户在价格接近买入区时提前关注，而不是等到价格真正进入区间后才被动反应。15 轮去重间隔。

### 7.8 动态买入区修正

买入区不是一成不变的。`_calc_dynamic_buy_zone()` 根据**三层联动因子**评估是否需要下移买入区。

**计算逻辑：**
1. 调用 `_get_market_adjustment(code, sector_trend)` 获取三层联动因子
2. 取 `buy_zone_shift`（买入区下移比例，范围 0~15%）
3. `new_min = buy_min × (1 - buy_zone_shift)`，`new_max = buy_max × (1 - buy_zone_shift)`
4. 修正幅度 < 2% → 静默使用，不告警
5. 修正幅度 ≥ 2% → 推送告警"买入区动态调整"，20 轮去重

**触发条件：** 大盘偏空 + 板块走弱时 `buy_zone_shift` 最大；大盘偏空但板块走强时 shift 减弱（三层联动）。

修正后的区间在 `_check_buy_candidates` 中**替代原区间**用于所有后续判断（区内/高于/低于）。

## 14. 买多少、怎么控制风险

### 8.1 仓位计算公式（`_calculate_position_size`）

```
最终金额 = 基准仓位 × 板块修正 × 买入区位置修正
（百元取整，下限 5000，上限 16000）
```

**第 1 步：确定基准仓位**

| 大盘状态 | 基准仓位 | 说明 |
|----------|----------|------|
| normal | 16000 | 大盘正常，满额 |
| v_reversal | 10000 | V 型反转确认中，半仓 |
| halt（大盘危险） | — | 仍然推送信号和操作建议，但不执行模拟盘买入 |
| panic / one_sided / dead_cat | 0 | 直接返回 0，不买 |

**第 2 步：板块趋势修正**

| 板块状态 | 修正系数 | 下限 |
|----------|----------|------|
| 持续走弱 | ×0.3 | 5000 |
| 走弱 | ×0.6 | 5000 |
| 持续走强 | ×1.3 | 上限 16000 |
| 走强 | ×1.2 | 上限 16000 |

**第 3 步：买入区位置修正**

| 价格在买入区的位置 | 修正系数 |
|-------------------|----------|
| 下沿 33% 以内 | ×1.1（激进） |
| 上沿 67% 以上 | ×0.7（保守） |

**大盘风险时的处理：** 当 `regime.allow_buy=False` 时，系统仍会推送买入信号到 Telegram，附带具体的市场状态和操作建议（如"恐慌加速下跌，不建议任何买入，已有持仓关注止损位"），但模拟盘不会自动执行买入。

### 8.2 风控引擎检查

在执行模拟盘买入前，还要过风控引擎（`RiskEngine.can_open()`），风控引擎通过 `set_regime()` 注入了 `MarketRegime` 对象，能感知大盘模式：

**检查 1：MarketRegime 仓位限制** — `regime.position_mult <= 0` 时直接拒绝开仓。

**检查 2：黑名单** — 如果股票在黑名单中，拒绝。

**检查 3：市场环境仓位上限**

| 市场环境 | 总仓位上限 |
|----------|-----------|
| bull（牛市） | 80% |
| swing（震荡） | 50% |
| bear（熊市） | 20% |

当前总仓位 + 新买入仓位 > 上限 → 拒绝。

**检查 4：集中度**

| 限制项 | 上限 | 含义 |
|--------|------|------|
| 单只股票 | 总资产的 20% | 不重仓一只票 |
| 单板块 | 总资产的 50% | 不押注一个行业 |
| 现金保留 | 总资产的 20% | 永不满仓 |

### 8.3 模拟盘交易执行

风控通过后，调用 `PaperAccount.buy()`：

1. 检查是否在连续竞价时间（9:30 之后）
2. 计算买入股数 = 金额 / 价格，取整到 100 股
3. 扣除佣金（万 0.85，最低 5 元，卖出时另收万 5 印花税）
4. 如果现金不够，按 90% 可用现金重新计算
5. 股数不足 100 → 放弃
6. 持仓已满 5 只 → 触发换仓评估
7. 执行买入：扣现金、建持仓、写 `trade_orders` 记录
8. 更新信号状态为 `bought`
9. 写入 `_pos_meta` 盯盘元数据（止损价、止盈价、移动止盈参数）
10. 推送到 Telegram

## 15. 怎么决定卖不卖

卖出管理在 `PositionRiskMixin`（position_risk.py）中。

### 9.1 五个卖出触发条件

每轮扫描检查所有持仓：

#### 类型 1：固定止损

- 触发条件：`现价 ≤ 止损价`
- T+1 保护：当日买入的不触发
- 执行：模拟盘自动卖出 + Telegram 推送 + 私聊确认请求（实盘）

#### 类型 2：固定止盈

- 触发条件：`现价 ≥ 止盈价`
- T+1 保护：当日买入的不触发
- 执行：同上

#### 类型 3：移动止盈

- 触发条件：`现价 ≤ 持仓期间最高价 × (1 - trailing_stop)`
- 默认 trailing_stop = 5%（即从最高点回撤 5% 就卖）
- 例子：持仓期间最高到过 100 元，trailing_stop = 0.05，触发价 = 100 × 0.95 = 95 元
- T+1 保护：当日买入的不触发

#### 类型 4：利润回撤止盈（分级）

这是最有特色的卖出规则。系统持续追踪每只持仓的**历史最高浮盈百分比**（`_pos_meta[code].max_profit_pct`），根据最高浮盈的级别，设置不同的保留比例：

| 级别 | 历史最高浮盈 | 保留利润比例 | 触发条件 |
|------|-------------|-------------|----------|
| T1 | ≥ 15% | 60% | 当前浮盈 < 最高浮盈 × 0.60 |
| T2 | ≥ 10% | 55% | 当前浮盈 < 最高浮盈 × 0.55 |
| T3 | ≥ 5% | 50% | 当前浮盈 < 最高浮盈 × 0.50 |

例子：某股最高浮盈到过 18%，现在回落到 9% → 9% < 18% × 0.60 = 10.8% → 触发 T1 回撤止盈。

如果最高浮盈没到过 5%，不触发利润回撤止盈。

#### 类型 5：时间止损

风控引擎中的时间止损：持有超 **5 天**仍在亏损 → 触发卖出信号。在 `check_positions` 的优先级 8 中检查。

### 9.2 止损止盈的统一处理流程

触发后 `_handle_stop_signal()` 执行：

1. **防重复**：如果该信号已在提醒队列中（`_sl_reminders`），跳过
2. **跌停保护**：如果跌停了，推送 `🚫 跌停无法卖出`，不执行，下一轮继续监控
3. **推送群聊**：告知触发类型、价格、盈亏
4. **推送私聊**：请求用户确认实盘操作（"回复「成交 XXX」确认"）
5. **加入提醒队列**：5 分钟后如果用户还没确认，再次推送
6. **模拟盘自动执行**：调用 `PaperAccount.sell()` 卖出（含 T+1 保护）
7. **清理盯盘状态**：从 `_pos_meta` 中移除

### 9.3 止损提醒循环

```
触发止损/止盈
  → 推送群聊 + 私聊
  → 加入 _sl_reminders 队列（status=pending, last_push=now）

每轮检查 _check_sl_reminders()：
  → 如果 status=pending 且 last_push 距今 > 300 秒（5 分钟）
  → 再次推送私聊提醒：「⏰ 再次提醒: XX信号 — 代码 名称」
  → 更新 last_push=now

用户回复处理：
  → "成交 002371" → 清理该代码的所有提醒
  → "再等 10 002371" → 将提醒设为 waiting，10 分钟后恢复
  → "再等 5" → 将所有提醒延迟 5 分钟
```

### 9.4 大盘/板块感知的动态调整

在检查止损止盈之前，`_check_positions` 会先读取大盘 `risk_level` 和板块趋势，动态调整止损/止盈/移动止盈的触发线：

**大盘环境调整因子：**

| risk_level | sl_tighten | tp_lower | trail_tighten | 含义 |
|-----------|------------|----------|---------------|------|
| safe | 1.0 | 1.0 | 1.0 | 不调整 |
| cautious | 0.92 | 1.0 | 0.92 | 止损略收紧 |
| dangerous | 0.85 | 0.90 | 0.85 | 止盈下调，止损收紧 |
| extreme | 0.70 | 0.80 | 0.70 | 大幅收紧，快速离场 |

**板块叠加修正（在个股级别叠加）：**
- 板块加速走弱：sl_tighten × 0.90，tp_lower × 0.90（额外收紧 10%）
- 板块走弱：sl_tighten × 0.95（额外收紧 5%）

**调整公式：**
```
effective_sl = avg_cost - (avg_cost - stop_loss) × sl_tighten
effective_tp = avg_cost + (take_profit - avg_cost) × tp_lower
effective_trail = highest_price × (1 - trailing_stop × trail_tighten)
```

effective_sl 不低于原止损的 85%，避免过度敏感。

### 9.5 利润回撤止盈的大盘加成

`_check_retracement_stop` 的保留比例会根据 `risk_level` 追加 bonus：

| risk_level | bonus | T1 保留(≥15%浮盈) | T2 保留(≥10%) | T3 保留(≥5%) |
|-----------|-------|-------------------|---------------|-------------|
| safe/cautious | 0 | 60% | 55% | 50% |
| dangerous | +0.05 | 65% | 60% | 55% |
| extreme | +0.10 | 70% | 65% | 60% |

大盘越危险，保留的利润越多，更早锁定收益。

### 9.6 持仓状态分类（6 级）

除了止损止盈，系统还会每 20 轮（约 20 分钟）或在状态变化时推送持仓状态。状态分 6 类，按亏损程度严格分层：

#### 1. healthy（健康）✅
- 条件：浮盈 > 2%
- 含义：持仓盈利，状态良好

#### 2. watching（观察）👀
- 条件：不满足其他分类
- 含义：正常波动范围内，继续观察

#### 3. at_risk（接近止损）🟠
- 条件：亏损 ≥ 2% 且已消耗止损空间 ≥ 85%
- 计算方式：(成本价 - 现价) / (成本价 - 止损价)
- 含义：距止损触发仅一步之遥，需做好离场准备

#### 4. trapped（被套）🔴
- 条件：浮亏 5% ~ 10%
- 含义：明显亏损，需评估反弹减仓机会
- 推送时附带三层离场分析和反弹目标价

#### 5. deep_trapped（深度套牢）💀
- 条件：浮亏 ≥ 10%
- 含义：严重亏损，建议立即评估是否割肉
- 推送时附带离场分析

#### 6. add_opportunity（补仓机会）🟡
- 条件：浮亏 ≥ 2% + 已消耗止损 < 50% + 布林 %B 在 5-30 + RSI12 < 40
- 含义：虽然亏损但在支撑位附近，可能有反弹机会
- 推送时附带详细的补仓分析（布林位置、RSI 状态）

### 9.7 被套离场时机分析（三层视角）

当状态为 trapped 或 deep_trapped 时，`_analyze_exit_context` 会从三层视角分析离场时机：

**第 1 层：大盘环境**
| 大盘状态 | 判断 |
|----------|------|
| 恐慌/极端 | 反弹不可靠，不建议等待，任何反弹都应减仓 |
| 危险 | 反弹空间受限，降低等待预期 |
| 谨慎 | 正常等待技术反弹 |

**第 2 层：板块走势**
| 板块状态 | 判断 |
|----------|------|
| 加速走弱 | 拖累个股，反弹力度有限，优先减仓 |
| 走弱 | 反弹可能受压制，不宜等太高 |
| 走强 | 可能带动个股反弹，可稍耐心 |

**第 3 层：个股技术指标**
- 接近布林中轨/MA60 阻力位 → 好的离场点
- MACD 空头排列 → 下跌趋势未止
- 远低于 BBI → 弱反弹即为减仓窗口
- RSI/KDJ 超卖 — 大盘恐慌或板块加速走弱时，超卖信号降级为"反弹不可靠"；正常环境才视为等待反弹信号

### 9.8 反弹减仓目标盯盘

对于 trapped/deep_trapped 持仓，`_calc_exit_target` 自动计算当前价上方最近的阻力位：

**候选阻力位（取最近）：** 布林中轨 > MA60 > BBI > 成本价

**盯盘逻辑：**
- 每轮扫描检查是否接近目标（距目标 ≤ 2% → 推送"准备减仓"，至少间隔 10 轮）
- 达到目标 → 推送"减仓目标达成"，自动计算下一个阻力位
- 板块加速走弱时，目标价下浮 2%（更保守）
- 状态恢复为非被套 → 自动清除目标

### 9.9 三层联动调整因子（大盘→板块→个股）

`_get_market_adjustment(code, sector_trend)` 是连接大盘情景引擎和个股决策的**统一接口**。它不是三个独立因子相乘，而是一体化联动评估：

**核心公式：**
```
大盘因子 = f(outlook.direction, outlook.urgency)
板块放大 = sector_trend 走弱 ? 1.2~1.4× : 走强 ? 0.4~0.6× : 1.0×
最终因子 = 大盘因子 × 板块放大
```

**三种联动模式：**

| 大盘 | 板块 | 联动效果 |
|------|------|----------|
| 偏空 | 走弱 | **共振放大** — 板块放大 1.2~1.4×，调整因子最大化 |
| 偏空 | 走强 | **减弱对冲** — 板块放大 0.4~0.6×，个股可能抵抗大盘 |
| 偏多 | 任意 | 正常/积极，不额外收紧 |

**输出因子（dict）：**

| 输出 | 范围 | 用途 |
|------|------|------|
| direction | bullish/bearish/neutral | 市场方向 |
| urgency | critical/act/watch/none | 紧急程度 |
| tp_ceil_factor | 0.4~1.0 | 止盈天花板折扣（<1 = 下调目标） |
| sl_tighten | 1.0~2.0 | 止损收紧比例（>1 = 收紧） |
| buy_zone_shift | 0~0.15 | 买入区下移比例 |

### 9.10 预测性止损止盈接近告警

`_check_predictive_proximity()` 在每轮持仓盯盘中运行，结合情景引擎预判**提前告警**，不等触发价才反应：

| 情况 | 条件 | 告警 |
|------|------|------|
| 接近止损+偏空 | 距 SL ≤ 3% + 市场偏空 + urgency ≥ act | ⚠️ 止损预警 |
| 接近止损（紧急） | 距 SL ≤ 1.5%（无论市场） | ⚠️ 接近止损 |
| 接近止盈+偏空 | 距 TP ≤ 3% + 市场偏空 | 📈 止盈预警（提前锁定） |
| 接近止盈（关注） | 距 TP ≤ 1.5% + urgency ≥ act | 📈 关注盘面 |

去重间隔 15 轮。目的是在触发前给用户心理准备和操作时间。

### 9.11 动态止盈止损目标修正

`_check_dynamic_targets()` 每轮评估持仓止盈/止损是否需要**动态修正**，不是一设不变。

**止盈天花板计算：**
1. `_find_resistance_ceiling(code, price)` 查最近技术阻力位（布林上轨/中轨/MA20/MA60/BBI）
2. 天花板 × tp_ceil_factor（三层联动因子）→ 修正止盈
3. 修正后 < 原止盈 × 0.97 → 建议下调 → 推送 🎯 告警

**止损收紧计算（cost 基准）：**
```
loss_width = entry_price - sl
tightened_width = loss_width × max(0.5, 2.0 - sl_tighten)
adjusted_sl = entry_price - tightened_width
adjusted_sl = max(adjusted_sl, 支撑地板 × 0.99)
```
4. 收紧 > 2% → 建议调整 → 推送告警

去重间隔 20 轮。当本次发送告警时，跳过同轮次的预测性接近告警，防止消息轰炸。

## 16. 异动检测和换仓

### 10.1 板块热度排名（每 50 轮）

调用独立的 `SectorHeatMonitor`，用全市场快照计算各板块平均涨跌幅，排名 TOP5。

会检查持仓/观察板块是否持续走弱或走强。

### 10.2 异动检测（每 3 轮）

用 `AbnormalDetector` 检测三种异动：

| 类型 | 阈值 | 含义 |
|------|------|------|
| 急速拉升 | 当前涨幅 - 上轮涨幅 > 1% | 盘中突然拉涨 |
| 量比暴增 | 当前量 > 上轮量 × 3 | 成交量异常放大 |
| 逼近涨停 | 涨幅 > 7% | 接近涨停板 |

### 10.3 主动换仓评估（每 15 轮）

当持仓 ≥ 3 只时触发。流程：

1. **收集候选**：从 pending 信号中筛出在买入区内或接近买入区（低于买入区 5% 以内）的
2. **构建上下文**：
   - 大盘背景（上证日内变动）
   - 板块行情（各行业/概念的实时涨跌）
   - 持仓实时涨跌
   - 候选信号实时涨跌
3. **AI 评估**（DeepSeek）：综合分析持仓和候选，判断是否该换仓
   - 回复 JSON：`{"sell": "要卖的代码", "buy": "要买的代码"}` 或 `{"sell": null, "buy": null}`
4. **AI 不可用时 fallback 规则**：
   - 优先卖「close」建议的（来自持仓审查）
   - 其次卖「reduce」建议的
   - 再次卖评分比新信号低 15 分以上的
5. **执行换仓**：先卖后买，T+1 保护（当日买入的不能被换出）

**换仓不会在以下情况触发：**
- 持仓 < 3 只
- 没有候选信号在买入区附近
- AI 决定不换仓
- 卖出的标的是当日买入的（T+1 锁定）

## 17. 尾盘怎么处理

14:30 后触发一次（`_check_closing`），对每只持仓判断是否适合持有过夜。尾盘决策会结合大盘环境和板块趋势做分级处理：

**大盘环境感知：**
- 大盘极端（extreme）→ 任何亏损都建议清仓，标题加 ⚠️ 大盘极端
- 大盘危险（dangerous）→ 浮盈 > 5% 时建议减仓锁定，微利也提示注意仓位

| 情况 | 建议 | 图标 |
|------|------|------|
| 今日买入 | 不可卖（T+1 锁定） | 🔒 |
| 大盘极端 + 浮亏 | 建议清仓，不持亏过夜 | 🚨 |
| 浮亏 > 3% | 建议尾盘止损，不持亏过夜 | 🔴 |
| 浮亏 1-3% 且距止损 < 3% | 关注是否触发止损 | 🟡 |
| 浮盈 > 5% + 大盘危险 | 利润可观+大盘风险，建议减仓锁定 | 🟢 |
| 浮盈 > 5% | 利润可观，可考虑减仓 | 🟢 |
| 大盘危险 + 微利 | 大盘风险较高，注意仓位 | ⚠️ |
| 其他 | 可持过夜 | ✅ |

只在有需要行动的情况时才推送消息。每只持仓显示板块趋势。

## 20. 数据存在哪、存多久

### 13.1 存储位置

所有数据存在 SQLite 文件：`/Users/biss/trading-system/storage/stock_market.db`

### 13.2 每张表的详细说明

#### market_snapshots（全市场原始快照）

- **用途**：保存每 3 轮约 5000 只股票的实时涨跌幅
- **字段**：`trade_date`（日期）、`ts`（时间戳精确到秒）、`code`（股票代码）、`change_pct`（涨跌幅，保留 4 位小数）
- **写入频率**：每 3 轮（约 3 分钟）
- **数据量**：每天约 40 万行，约 10MB
- **清理策略**：启动时自动删除 3 天前的数据
- **为什么存**：盘中重启时可以恢复全天板块趋势

#### sector_snapshots（板块聚合快照）

- **用途**：保存每轮的板块平均涨跌幅
- **字段**：`trade_date`、`ts`、`sector_name`（板块名）、`avg_change`（平均涨跌幅）、`up_count`（涨家数）、`down_count`（跌家数）、`market_avg`（全市场均值）
- **写入频率**：每 3 轮
- **清理策略**：启动时自动删除 3 天前的数据

#### trade_signals（交易信号）

- **用途**：所有买入候选信号
- **状态流转**：`pending` → `bought`（已买入）或 `expired`（过期）
- **清理策略**：收盘时 pending → expired（当天没有触发的信号自动过期）
- **保留时间**：永久（bought 和 expired 都保留，用于回测分析）

#### trade_orders（交易订单）

- **用途**：记录每一笔模拟/实盘交易
- **保留时间**：永久
- **用途**：启动时恢复持仓（查所有 buy 订单的净持仓）

#### trade_portfolio_snapshots（账户日终快照）

- **用途**：每天收盘时的账户总览
- **字段**：`total_value`、`cash`、`market_value`、`daily_pnl`、`total_pnl`、`drawdown`、`position_count`
- **保留时间**：永久
- **用途**：追踪最大回撤（启动时取历史最高 total_value）

#### trade_portfolio_positions（持仓明细日终快照）

- **用途**：每天收盘时每只持仓的详细信息
- **保留时间**：每日覆盖

#### trade_holdings_review（持仓审查建议）

- **用途**：AI 对每只持仓的审查结果
- **字段**：`action`（close/reduce/hold）、`new_stop_loss`、`new_take_profit`、`tomorrow_outlook`、`reason`
- **用途**：换仓评估时作为参考（优先卖出建议 close 的持仓）

#### stock_basic / stock_indicators

- **数据来源**：盘前数据采集流程（非 watcher 负责）
- **保留时间**：stock_basic 保留 120 天（用于 RPS 计算）
- **用途**：提供均线、布林带、RSI、MACD、KDJ、主力资金等日线级别数据

### 13.3 清理策略汇总

## 23. 策略审计系统

### 19.1 它是干嘛的

策略审计系统（`python main.py strategy-audit`）定期检查策略管线的决策质量，发现系统性偏差，生成改进建议。它是策略的"自检医生"，不是给人看的报表，而是让系统自己发现问题。

### 19.2 双轨审计

**规则审计（RuleAuditor）：**
基于硬规则的统计分析，检查策略漏斗和 AI 决策的历史数据：
- 漏选检查：AI 淘汰了但第二天涨停的股票有多少？
- 错选检查：AI 入选了但第二天大跌的股票有多少？
- 因子有效性：哪些筛选因子的通过率和次日表现最不匹配？
- 场景偏移：AI 在哪些场景下系统性地过于乐观/悲观？

**AI 审计（AIAuditor）：**
把规则审计的发现 + 历史决策数据发给 AI，让它从更高维度分析：
- 偏见检测：AI 是否有某种系统性偏好（如偏好某个行业、某个技术形态）？
- 遗漏模式：是否存在一批股票被反复淘汰但其实表现不错的规律？
- 改进建议：具体到参数调整和代码修改的建议

### 19.3 数据流

```
读：strategy_funnel + strategy_ai_decisions（历史决策数据）
    strategy_lessons（已有教训，避免重复）

写：strategy_lessons（新发现的教训）
    strategy_improvements（改进建议，状态 pending）
```

### 19.4 改进应用

`strategy-audit --apply N` 可以自动应用第 N 条 pending 改进建议。`improvement_applier.py` 负责执行，应用后记录 `applied_date`，后续可跟踪 `effectiveness_check`。
