## 认可、应该修的

| 编号 | 问题 | 理由 |
|------|------|------|
| **B3** | `trade_holdings_review` 表未在 schema.py 定义 | 确实漏了。repo.py 操作该表但不创建，DB 里现有的表可能是旧版或手动建的。`ensure_tables()` 没有定义，首次部署会报错。 |
| **F4** | `_bought_watch` 卖出后不清理 | 内存泄漏。买入后添加 `_bought_watch[code]`，但 `close()` 和平仓路径都没有清理。长期运行字典持续增长，已卖出股票的数据毫无意义。 |
| **F5** | `_get_watch_codes` 每轮查 DB | 每轮扫描都调 `repo.get_pending_signals()` 和 `_load_review_picks()`。实际分析：positions 来自内存不需要查 DB；signals 盘中极少变化（可能一整天不变）；review_picks 也来自 DB 查询。缓存策略：signals + review_picks 的 code 列表缓存，刷新时机为①模拟盘成交后（买卖都会改变关注列表）②收盘时。 |
| **K1** | HTML 转义缺 `&` | 正确的 bug。Telegram HTML parse mode 要求 `&` → `&amp;` 必须最先转义。如果消息里有 `&`（比如公司名"T&C科技"），Telegram 会解析失败导致整条消息发不出去。 |
| **K1** | HTML 转义缺 `&` | 正确的 bug。Telegram HTML parse mode 要求 `&` → `&amp;` 必须最先转义。如果消息里有 `&`（比如公司名"T&C科技"），Telegram 会解析失败导致整条消息发不出去。 |
| **H1** | 印花税注释歧义 | `# 万分之五（减半征收）` → 应改为 `# 万分之五（2023年8月减半后标准）`，明确说明当前实际税率。 |
| **W1** | 缺少大盘量能监控 | 好建议。量价背离（价升量缩=诱多）是判断行情持续性的核心指标。数据已经有了——QMT `/all_quotes` 返回全市场数据，可以在 `_check_market_state` 中增加成交量对比（vs 昨日同时段）。 |
| **W9** | 缺少尾盘决策 | 14:30 后是短线交易关键时段。至少应该做：① 持仓是否过夜判断（浮亏票尾盘是否止损）；② 尾盘异动提醒（放量拉升/跳水）。可以在 `_check_market_state` 或新增 `_check_closing()` 中实现。 |
| **R2** | 缺少最大回撤保护 | 模拟盘 20 万需要总回撤上限。建议设 15%，触及时暂停所有买入、推送告警、所有持仓建议减仓。和现有的日内熔断（3%）互补——日内的是单日保护，这个是累计保护。 |

## 不同意、不应该修的

| 编号 | 问题 | 理由 |
|------|------|------|
| **A1** | `return False` 后面 `raise` 死代码 | 这段代码已经不存在了，审计报告基于旧版本。 |
| **B1/B2** | 索引缺失 | 信号表几千条、订单表几百条、全表扫描在 SQLite 上毫秒级完成。在当前数据规模下加索引是过早优化，增加了维护负担（schema.py 多几十行 DDL）但实际性能提升感知不到。 |
| **C1** | 频繁创建 SQLite 连接 | 同上。当前数据量级下连接开销可以忽略。连接池反而增加复杂度（需要管理生命周期、线程安全）。 |
| **E1/F1** | 文件过大要拆分 | 用户明确偏好不做过度抽象。`watcher.py` 虽然 1200 行但职责单一——盯盘。拆成 market_monitor / position_monitor / signal_trigger / swap_evaluator 四个文件后，看一个完整的扫描流程需要跳 4 个文件，理解和维护成本反而更高。 |
| **F2** | `_check_positions` 和 `_check_bought_signals` 重复 | 两处数据源不同：前者来自 `portfolio.positions`（内存中的模拟盘持仓），后者来自 `trade_signals WHERE status='bought'`（DB 中的实盘信号）。虽然逻辑相似，但合并会引入耦合——需要同时传内存和 DB 数据，且两边数据可能不一致（比如用户手动录入了但模拟盘没买）。分开维护虽然有几行重复代码，但职责边界清晰。 |
| **F3** | AI 指数波动无冷却 | 已经有 `_index_last_fluctuation_price` 防重复推送——价格偏离上次预警超过 0.3% 才重新调 AI。不需要额外的时间冷却。 |
| **G2** | `open_position` 不检查重复 | `PaperTrader.try_buy()` 第一行就做了 `if code in self.portfolio.positions: return False`。portfolio 层的 `open_position` 被设计为接受覆盖（用于恢复持仓），不需要再加检查。 |
| **H3** | 硬编码 DeepSeek 端点 | 目前只有一个模型做换仓评估（deepseek-chat），没有切换需求。加配置项是提前抽象——真的需要换模型时再加。 |
| **H4** | `MAX(trade_date)` 可能过时 | `_load_reviews` 是规则兜底（AI 失败后才走），本身就低频使用。如果说 AI 连续几天失败导致 review 过期——那说明 AI 本身有问题，应该先修 AI。 |
| **W5** | 板块热度 10 分钟太慢 | 板块热度变化没那么快，10 分钟间隔足够。缩短到 2-3 分钟只是多消耗 QMT `/all_quotes` API（该端点一次 4 秒），收益极低。 |
| **P1** | 筛选 20 只太多 | 经过硬关卡过滤 + 安全网验证后，实际到达 AI 的候选通常 10-15 只。而且 AI 需要有一定选择空间——如果只给 5 只，AI 只能"矮子里拔高个"。 |
| **P3** | 板块 gate 过度过滤 | 设计如此。不属于近期热点板块的独立行情通常是一日游，不可持续。过滤掉是正确的风险控制。如果真的出现新热点，sector_hot_history 通常会在一两天内收录。 |

## 值得讨论、不完全同意

| 编号 | 问题 | 我的看法 |
|------|------|----------|
| **G1** | daily_pnl 语义 | 报告说得对——`snapshot()` 会重置 `_prev_total`。实际分析：`_prev_total` 初始值是 `initial_cash`（20 万），盘中不调 snapshot 时，`daily_pnl` 算的是相对初始资金的变化而非相对开盘价。有大额出入金时偏差明显。**可以修但优先级不高——需要记录开盘时的 total_value 作为基准。** |
| **J1/R1** | 黑名单为空 | ST 和 688 在 `check_hard_gates` 中已过滤。被立案但还没 ST 的票——这种黑名单需要持续人工维护（每月都有新立案的），且 AI 在分析时会综合监管函和电报利空来判断。**不需要代码级永久黑名单，但如果用户有特别不想碰的票，可以从 DB 维护一个黑名单表。** |
| **W7** | 止损分级（龙头 vs 跟风） | 方向对——不同角色的票止损策略应不同。但不需要硬编码规则（"龙头放宽 10%，跟风严格 5%"这种太机械）。**AI 持仓审查已经在做这件事**——每只票的止损止盈是 AI 根据个股走势、技术位、板块强度独立给出的，比硬编码灵活得多。 |
| **W3** | 盘中新闻实时监控 | 方向对，但电报已经在每 5 分钟 cron 采集（`collect --module news`）。实时推送需要设计关键词触发机制。**可以做轻量版：Watcher 每 N 轮扫最近 10 分钟的电报，匹配关键词则推送。** 关键词不应太窄，至少包含：立案调查、业绩暴雷、减持、实控人变更、重大合同终止、被监管问询、重大诉讼、ST 风险警示。 |
| **W4** | 涨停板实时监控 | 需要 QMT 的 `limit_pool` 数据支持（封单金额、炸板实时）。目前有涨跌停价缓存（`_limit_cache`），但没有封单数据，QMT 权限也不支持。**等 QMT 策略交易权限开通后再说。** |
| **F6** | `_index_prices` 60 点够不够 | 60 点 × 60s = 1 小时。对于 `_classify_market_pattern` 的三段分析（每段 ≥ 10 点）来说够了。如果 scan_interval 改成更小的值，可以调整这个上限。**暂时不改。** |

## 对审计报告的总体评价

报告覆盖面广、格式专业、两个视角互补。架构师视角偏工程规范（"如何不犯错"），游资视角偏交易实战（"如何赚到钱"），都在点子上。

但部分"严重"问题在当前项目规模下实际影响很小（如 SQLite 索引），部分"代码重复"是刻意为之（数据源不同不应合并）。审计标准偏向"大团队协作项目的工程规范"，而当前项目的约束是"一人维护、快速迭代、不做过度抽象"——有些规范不适用。

**最有价值的建议**集中在游资操盘手部分：大盘量能、尾盘决策、最大回撤保护——这些是真正能提升交易质量的。
