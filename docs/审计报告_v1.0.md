### 架构概览
```
trading-system/
├── main.py              # CLI 入口（13 个命令）
```

### 核心数据流
```
盘后复盘: 数据采集(14个) → AI分析(FC多轮) → 报告推送 → 股票追踪
盘前管线: 市场宽度 → 趋势筛选 → 画像富化 → AI精选 → 信号入库 → 推送
盘中盯盘: 实时行情 → 三层扫描(风控/信号/异动) → 自动交易/提醒
```

### 一、系统架构评价：⭐⭐⭐⭐ (4/5)

#### ✅ 架构亮点

1. **分层清晰**：数据层(data) → 分析层(analysis) → 交易层(trade) → 基础设施(system)，职责分明
2. **采集器独立**：14 个采集器各自 try/except 隔离，单一失败不影响整体
3. **SQLite 选型合理**：单机交易系统，SQLite 完全够用，性能足够
4. **幂等设计**：表结构创建用 `IF NOT EXISTS`，ALTER TABLE 用 try/except 做迁移
5. **PID 文件防多实例**：盯盘进程用 PID 文件防止重复启动
6. **信号状态机**：pending → bought / expired / rejected，状态流转清晰
7. **实盘/模拟盘分离**：通过 account 字段区分，持仓查询和快照独立统计

### 二、代码质量审计：逐模块细查

#### 2.1 `main.py` — CLI 入口

**问题 A1 (严重) — 死代码**
```python
# main.py 第 278 行，cmd_review 的 generate_and_send 末尾
return False
raise  # ← 这行永远不会执行！return 后面 raise 是死代码
```
**修复**: 删除最后一行 `raise`

**问题 A2 (中等) — 命令分发无错误处理**
```python
# main.py main() 函数
{...}.get(cmd, lambda: print(f"Unknown: {cmd}"))()
```
未知命令只 print，返回码仍是 0，不利于 cron 判断执行成功/失败。
**建议**: 未知命令应 `sys.exit(1)`

**问题 A3 (轻微) — scan_interval=0**
```python
watcher = Watcher(..., scan_interval=0, ...)
```
scan_interval=0 意味着扫描间无间隔，完全依赖 scan 内部的 time.sleep 逻辑。虽然 watcher.py 内部控制了间隔，但传 0 作为参数不直观。

#### 2.2 `data/schema.py` — 数据库表结构

**问题 B1 (严重) — trade_orders 缺少 account 字段的索引**
```sql
-- trade_orders 表没有 account 字段
-- 但 repo.py 中 get_orders_by_date 用 WHERE trade_date=? AND account=? 查询
```
**影响**: 大量订单时查询慢，尤其在 compare（收盘比对）和 strategy（加载持仓）中频繁查询。
**修复**: 添加索引 `CREATE INDEX idx_trade_orders_date_account ON trade_orders(trade_date, account)`

**问题 B2 (严重) — trade_signals 缺少关键索引**
```python
# repo.py get_expired_signals 使用:
WHERE status='expired' AND trade_date < ? AND strategy_name LIKE 'ai_advisor%'
```
没有 `(status, trade_date, strategy_name)` 复合索引，过期信号查询会很慢。

**问题 B3 (中等) — trade_holdings_review 表未在 schema.py 中创建**
`repo.py` 中有 `insert_holdings_review()` 和 `apply_holdings_review_sl_tp()` 方法，操作 `trade_holdings_review` 表，但该表不在 `ensure_tables()` 中定义。如果表不存在，首次运行会报错。
**修复**: 在 schema.py 的 `ensure_tables()` 中添加该表定义。

**问题 B4 (中等) — cls_telegraph ALTER TABLE 迁移可能遗漏**
schema.py 对 cls_telegraph 表做 ALTER TABLE ADD COLUMN 迁移，但如果该表由其他采集器首次创建，可能存在竞态条件（采集器先创建表，但缺少新列，迁移失败后静默跳过）。

**建议**: 所有表的列定义应集中在 schema.py，采集器只负责写数据。

#### 2.3 `data/repo.py` — 数据库访问层

**问题 C1 (严重) — 频繁创建/关闭连接**
每个方法都 `conn = self._conn()` ... `conn.close()`，在 watcher 每轮扫描中会创建数十个连接。虽然 SQLite 是文件级锁，但频繁 open/close 有性能损耗。
**建议**: 使用连接池或在 Watcher 级别复用连接。

**问题 C2 (中等) — insert_positions 未使用事务**
```python
def insert_positions(...):
    conn.execute("DELETE FROM ...")  # 先删
    for p in positions:
        conn.execute(f"INSERT INTO ...")  # 后插
    conn.commit()  # 一起提交
```
虽然最终 commit 了，但如果中间某条 INSERT 异常，会导致部分删除、部分插入的不一致状态。
**修复**: 用 `try/except` 包裹，异常时 rollback。

**问题 C3 (中等) — get_pending_signals 列名硬编码**
```python
cols = ["id", "trade_date", ..., "account", "expected_trend"]
```
如果数据库表增加了新列（如 `expected_trend`），这里的硬编码列名列表不会自动更新，可能导致 index mismatch。
**修复**: 使用 `conn.row_factory = sqlite3.Row` 后用字典访问，或动态获取列名。

**问题 C4 (轻微) — insert_order 的 SQL 用 f-string 拼接列名**
```python
sql = f"INSERT INTO trade_orders ({cols}) VALUES ({placeholders})"
```
虽然列名来自内部 dict 可控，但如果未来有外部输入作为 key，存在 SQL 注入风险。建议用白名单校验列名。

#### 2.5 `analysis/review/analyzer.py` — AI 分析器

**问题 E1 (严重) — generate() 函数超过 600 行**
单个函数做数据采集、格式化、AI 调用、报告解析等所有事情，违反单一职责原则。
**建议**: 拆分为 `_load_market_data()`, `_build_prompt()`, `_call_ai_fc()`, `_parse_results()` 等方法。

**问题 E2 (严重) — SQL 注入风险（f-string 拼接 IN 子句）**
```python
placeholders = ','.join('?' * len(index_codes))
# 这部分是安全的（用了 ? 占位符）
# 但：
f"INSERT OR REPLACE INTO trade_holdings_review ({cols}) VALUES ({placeholders})"
# cols 来自 dict.keys()，如果是外部输入则不安全
```
当前代码中 cols 来自内部构造，暂时安全，但缺乏防护机制。

**问题 E3 (中等) — FC 日志文件可能过大**
`_run_fc_review()` 中每轮都记录完整工具调用和返回内容，当 AI 多次调用工具时，日志文件可能非常大。
**建议**: 对工具返回结果做截断（已做 500 字截断），但考虑增加最大日志轮数限制。

**问题 E4 (轻微) — `telegraph_skip_keywords` 可能误过滤**
```python
TELEGRAPH_SKIP_KEYWORDS = ['三大指数', '沪深两市成交额突破', ...]
```
"三大指数"可能在某些重要电报中出现，导致被误过滤。

#### 2.6 `trade/monitor/watcher.py` — 盘中盯盘（1200+ 行）

**问题 F1 (严重) — 文件过大，违反单一职责**
1200+ 行的单文件，包含市场状态检测、持仓风控、信号触发、异动检测、板块热度、换仓评估、Telegram 消息接收等所有功能。
**建议**: 拆分为 `market_monitor.py`, `position_monitor.py`, `signal_trigger.py`, `swap_evaluator.py` 等模块。

**问题 F2 (严重) — `_check_positions` 和 `_check_bought_signals` 逻辑重复**
两处都有相同的 T+1 判断、止损/止盈/利润回撤逻辑，如果修改一处容易遗漏另一处。
**建议**: 抽取为 `_check_single_position(code, price, entry_price, sl, tp, is_today_buy)` 公共方法。

**问题 F3 (中等) — 指数波动预警中直接调用 AI**
```python
def _analyze_index_fluctuation(self):
    # ... 构建 prompt
    result = ai._call_ai(prompt, system_prompt=..., max_tokens=300)
```
每轮扫描波动 ≥ 0.5% 时都调用 AI，如果市场剧烈波动可能频繁触发 AI 调用，产生大量 API 费用且可能触发 rate limit。
**建议**: 增加冷却时间（如 5 分钟内不重复调用），或用规则预判替代部分 AI 调用。

**问题 F4 (中等) — `_bought_watch` 状态清理缺失**
```python
self._bought_watch[code] = {...}  # 买入后添加
```
但卖出后没有清理 `_bought_watch[code]`，导致字典持续增长。长期运行可能累积大量已卖出股票的无效数据。
**修复**: 在 `close()` 或卖出信号处理时清理 `_bought_watch` 中对应条目。

**问题 F5 (中等) — `_get_watch_codes` 每轮都查 DB**
每轮扫描都调用 `repo.get_pending_signals()` 和 `_load_review_picks()`，数据库查询频繁。
**建议**: 缓存 watch_codes，只在特定事件（新信号入库、收盘）时刷新。

**问题 F6 (轻微) — `_index_prices` 列表增长控制**
```python
if len(self._index_prices) > 60:
    self._index_prices = self._index_prices[-60:]
```
60 个价格点如果 scan_interval=60 秒，约等于 1 小时的数据，用于判断日内走势可能不够。但如果是 scan_interval 更小（如 10-30 秒），60 个点也够用。

**问题 F7 (轻微) — 午休判断用 `time.sleep(30)` 轮询**
```python
@staticmethod
def _lunch_break():
    while Watcher._in_lunch_break():
        time.sleep(30)
```
不如直接计算到 13:00 的秒数然后 sleep 一次。

#### 2.7 `trade/portfolio/portfolio.py` — 组合管理

**问题 G1 (严重) — daily_pnl 和 snapshot 语义不一致**
```python
@property
def daily_pnl(self) -> float:
    return self.total_value - self._prev_total
```
`_prev_total` 在 `snapshot()` 中被更新为当前值。如果盘中多次调用 snapshot()，daily_pnl 会被重置。这意味着 daily_pnl 反映的是"上次 snapshot 以来的变化"，而非"今日开盘以来的变化"。
**影响**: 风控引擎中的日内熔断检查（`check_daily_loss_limit`）可能不正确。
**修复**: 应记录开盘时（或首个 snapshot 时）的 value，daily_pnl 始终基于开盘 value 计算。

**问题 G2 (中等) — `open_position` 不检查重复持仓**
```python
def open_position(self, stock_code, ...):
    self.positions[stock_code] = pos  # 直接覆盖！
```
如果同一股票被重复买入，旧持仓被静默覆盖，导致盈亏计算错误。
**修复**: 在 open_position 前检查 `if stock_code in self.positions: return False`

**问题 G3 (轻微) — Position 缺少 `entry_price` 字段**
`avg_cost` 考虑了佣金，但有些场景（如计算滑点）需要原始成交价。

#### 2.11 `system/utils/telegram.py` — Telegram 工具

**问题 K1 (严重) — HTML 转义不完整**
```python
escaped_chunk = chunk_text.replace('<', '&lt;').replace('>', '&gt;')
```
Telegram HTML 模式还需要转义 `&` → `&amp;`。如果不转义 `&`，当消息中包含 `&` 时 Telegram 会解析失败。
**修复**:
```python
escaped_chunk = (chunk_text
    .replace('&', '&amp;')
    .replace('<', '&lt;')
    .replace('>', '&gt;'))
```

**问题 K2 (中等) — `verify=False` 禁用 SSL 验证**
```python
resp = requests.post(url, ..., verify=False, ...)
```
在生产环境中禁用 SSL 验证是安全隐患。虽然是为了代理环境工作，但建议至少支持配置。

**问题 K3 (轻微) — MessageReceiver 的 `fetch_updates` timeout 参数**
默认 timeout=10 秒，但 HTTP 请求 timeout=timeout+10=20 秒。如果网络慢，可能阻塞盯盘扫描。

#### 2.13 `analysis/screening/factors.py` — 筛选因子

**问题 M1 (中等) — `check_hard_gates` 中科创板判断只检查前缀**
```python
if code.startswith("688"):  # 科创板
    return False
```
这只检查了 688 前缀，但创业板 300/301 新股（N/C 前缀）的检查在黑名单模块中，不在 hard_gates 中。两处检查分散。

**问题 M2 (轻微) — `check_trend_strength` 中 sharpe 计算公式**
```python
sharpe = (mean_ret / std) * (252 ** 0.5)
```
这里用的是日收益率，乘以 √252 年化是正确的。但 mean_ret 是百分比值（如 0.5 表示 0.5%），不是小数（0.005），所以 sharpe 值会偏大 100 倍。不过由于只判断 > 1.0，且所有股票用同样公式，相对比较不受影响。

## 🐉 第二部分：顶级游资操盘手审计

### 一、复盘任务设计审计

#### ✅ 做得好的地方

1. **数据采集全面**：14 个采集器覆盖行业板块、概念板块、个股行情、强势股、龙虎榜、涨跌停、监管函、大盘指数、股票异动、停复牌、股东增减持、公告、隔夜宏观、CLS复盘新闻
2. **AI 多轮 FC 工具调用**：不是简单给数据让 AI 输出，而是让 AI 主动调用工具获取细节数据（电报、龙虎榜席位、昨日推荐表现等），模拟真实操盘手的研究过程
3. **昨日复盘自我校准**：每次复盘加载昨日报告，对比"昨日预判 vs 今日实际"，AI 能自我修正
4. **连板晋级率统计**：近 3 日连板晋级率变化，判断情绪周期
5. **涨停质量分析**：一字板/换手板/回封板分类，区分封板质量
6. **昨日涨停今日表现**：跟踪昨日涨停股今日溢价率，判断接力情绪
7. **仓位硬顶机制**：基于涨停数、封板率、最高板、涨跌比、溢价率综合计算仓位上限
8. **风险地雷**：股东增减持 + 重点监控 + 重要公告 + 避雷针交叉匹配

#### ❌ 不足与改进建议

**问题 T1 (严重) — 缺少集合竞价数据采集**
复盘是盘后做的，但集合竞价（9:15-9:25）的数据对次日预判极其重要。
**建议**: 增加集合竞价数据采集（竞价金额/竞价涨幅/竞价封单），作为早盘简报的输入。

**问题 T2 (严重) — 龙虎榜席位分析不够深**
当前只展示了席位明细（买入/卖出金额、是否机构/游资），但缺少：
- 席位关联分析（哪些席位经常一起操作某只票）
- 席位历史胜率统计
- 顶级游资（如赵老哥、章盟主）的专属标记
- 机构专用 vs 游资的博弈分析

**问题 T3 (中等) — 缺少板块轮动预判**
复盘报告中判断了"绝对主线"和"退潮方向"，但缺少：
- 主线板块的生命周期判断（启动/发酵/高潮/退潮哪个阶段）
- 资金从退潮方向流出后可能流入的承接方向预判
- 跨日板块热度趋势（连续几天上榜的板块 vs 一日游）

**问题 T4 (中等) — 炸板分析不够细**
当前只统计了炸板数量和明细，但缺少：
- 炸板时间点分析（早盘炸板 vs 尾盘炸板，意义完全不同）
- 炸板后回封率
- 炸板股的次日表现统计

**问题 T5 (轻微) — 缺少大宗交易/北向资金数据**
复盘数据中没有大宗交易数据和北向资金流向（虽然在 collectors 中有 macro_collector，但北向资金没有单独采集）。

### 二、盯盘交易模块审计

#### ✅ 做得好的地方

1. **三层扫描设计合理**：
   - 第一层（每轮）：大盘状态 + 持仓风控 + 信号触发
   - 第二层（每50轮）：板块热度
   - 第三层（每3轮）：异动检测
2. **大盘智能分层**：panic/one_sided/dead_cat/v_reversal/normal 五种模式，比简单的"涨/跌"判断更精准
3. **T+1 锁定**：今日买入的股票不触发止损止盈，符合 A 股规则
4. **分级利润回撤止盈**：浮盈≥15% 保留60%利润，≥10% 保留55%，≥5% 保留50%，合理
5. **跌停保护**：跌停时不触发止损，等下一轮
6. **板块集中度预警**：同一板块信号≥3只时提示分散风险
7. **智能仓位计算**：结合大盘模式、板块趋势、买入区位置动态调整买入金额
8. **止损提醒循环**：推送后5分钟未确认再推，避免用户漏看
9. **买入后盯盘**：每50轮推送持仓状态（健康/观察/被套/补仓机会）
10. **主动换仓评估**：每15分钟评估是否换仓，AI+规则双重判断

#### ❌ 不足与改进建议

**问题 W1 (严重) — 缺少大盘量监控**
大盘缩量/放量是判断行情持续性的重要指标，当前只监控了价格走势，没有监控：
- 上证成交量（对比昨日同时段放量/缩量）
- 两市成交额（万亿是重要心理关口）
- 量价背离（价升量缩 = 诱多风险）

**问题 W2 (严重) — 集合竞价阶段只做汇总，不做验证**
开盘决策汇总在 `scan_count == 1` 时推送，但没有：
- 验证信号股竞价情况（竞价高开/低开/竞价金额）
- 竞价涨停/跌停统计
- 竞价异常（大幅高开但竞价量小 = 诱多）

**问题 W3 (严重) — 盘中缺少新闻实时监控**
盘中突发事件（政策利好/利空、公司突发公告）对短线交易影响巨大，但当前只在复盘和早报时采集新闻。
**建议**: 盘中电报实时监控（已采集但未实时监控），设置关键词触发即时推送。

**问题 W4 (严重) — 缺少涨停板实时监控**
涨停板是短线交易的核心标的，当前缺少：
- 涨停板封单金额实时监控
- 炸板即时推送
- 涨停板排队买入机会（虽然 A 股散户很难排队，但可以作为观察指标）
- 昨日涨停今日连板率（盘中实时更新）

**问题 W5 (中等) — 板块热度扫描频率太低**
每 50 轮（约10分钟）扫描一次板块热度，对于快速轮动的行情来说太慢。
**建议**: 提高到每 10-15 轮（2-3 分钟）扫描一次。

**问题 W6 (中等) — 缺少分时图形态识别**
当前技术指标主要基于日线和分钟级 MACD/RSI/KDJ，但缺少：
- 分时图形态（如 W 底、头肩顶、V 型反转）
- 分时均线（黄线）支撑/压力判断
- 尾盘拉升/砸盘识别

**问题 W7 (中等) — 持仓止损/止盈没有分级**
当前所有持仓用相同的止损止盈逻辑，但不同角色的票应该有不同的止损策略：
- 龙头股：放宽止损，给更多空间
- 跟风股：严格止损，破位就走
- 中线趋势股：用 MA20 止损
- 短线突破股：用 MA5/MA10 止损

**问题 W8 (中等) — 模拟盘自动交易缺乏滑点处理**
```python
# PaperTrader try_buy()
volume = int(capital / price / 100) * 100
```
直接用当前价格计算，没有考虑实际成交可能有滑点。虽然模拟盘不需要精确，但如果滑点过大，模拟结果会失真。

**问题 W9 (中等) — 缺少尾盘决策**
14:30 之后是短线交易的关键时段，但当前没有专门的尾盘决策逻辑：
- 持仓是否过夜判断
- 尾盘买入机会（尾盘放量突破）
- 尾盘止损（日内亏损股尾盘必须处理）

**问题 W10 (轻微) — 缺少仓位统计面板**
盯盘过程中没有实时显示当前仓位情况（总资产/现金/持仓比例/各股占比），用户需要额外查询。

### 三、策略管线审计

#### ✅ 做得好的地方

1. **多因子筛选**：20+ 因子覆盖量价、资金、趋势、RPS、板块共振
2. **硬关卡过滤**：先过 6 道硬关卡（ST/科创板/涨跌停/市值/量比/均线排列），减少无效计算
3. **板块黑名单**：硬编码排除白酒、银行、保险、地产等不适合趋势交易的板块
4. **严重风险过滤**：监管函（财务造假/信披违规）+ 电报利空（业绩暴雷/减持/诉讼）一票否决
5. **复盘上下文注入**：将复盘报告的情绪周期、主线方向、仓位建议注入策略管线 AI prompt
6. **昨日遗留推荐**：未买入的 AI 信号次日重新评估
7. **安全网验证**：AI 返回的信号重新用硬关卡验证，防止 AI 幻觉
8. **持仓审查**：AI 对所有持仓逐一审查，给出 hold/add/reduce/close 建议
9. **趋势定级和买点建议**：信号推送时附带趋势定级、理想买点、止损位、持仓周期

#### ❌ 不足与改进建议

**问题 P1 (严重) — 筛选结果上限过高**
```python
return candidates[:self.top_n * 2]  # top_n=10, 返回20只
```
20 只候选给 AI 分析，prompt 会非常长，AI 可能无法对每只票做深入分析。
**建议**: 减少到 10-12 只，或者分批次给 AI 分析。

**问题 P2 (中等) — 因子打分过于简单**
```python
def _compute_score(self, tags, scenarios, row):
    base = 20.0
    base += len(tags) * 5       # 每个因子 +5
    base += len(scenarios) * 8  # 每个场景 +8
    base += angle * 2           # MA5角度 * 2
```
所有因子权重相同，但实际上不同因子的含金量不同：
- "主力介入" + "筹码集中" 比 "缩量回调" 更有价值
- "RPS多周期共振" 比单一 RPS20 强更有说服力
- "周线多头" 是中期趋势确认，应该给更高权重

**问题 P3 (中等) — 板块 gate 可能过度过滤**
```python
def _check_sector_gate(row, sector_hot, stock_sectors):
    if not sector_hot:
        return True  # 板块数据为空时放行
    # 所属板块必须至少有一个在近10日热点榜上
```
如果板块数据缺失（DB 为空），所有股票都放行；如果有数据，不属于热点板块的股票全部过滤。这种"非黑即白"的 gate 可能错过：
- 即将启动的新热点（还没上榜）
- 冷门板块的独立行情

**问题 P4 (轻微) — 缺少基本面因子**
虽然有 PE/PB/营收增长/利润增长数据，但在筛选因子中没有使用基本面因子。趋势交易虽然以技术面为主，但基本面太差（如亏损股）的票风险更大。

### 四、风控体系审计

#### ✅ 做得好的地方

1. **多层风控**：黑名单 → 市场环境 → 集中度 → 止损 → 止盈 → 日内熔断 → 时间止损
2. **市场环境动态仓位**：牛市 80% / 震荡 50% / 熊市 20%
3. **单票/单板块集中度限制**：单票 20%，单板块 30%
4. **分级利润回撤止盈**：根据最高浮盈动态调整回撤容忍度
5. **恐慌模式全部停止**：大盘恐慌时暂停所有买入，建议减仓
6. **板块集中度预警**：同一板块信号过多时提示

#### ❌ 不足与改进建议

**问题 R1 (严重) — 黑名单实际为空**
`PERMANENT_BLACKLIST = set()` 等于没有黑名单。虽然有 `check_hard_gates` 过滤 ST 和 688，但那些不在硬关卡中的风险股（如被监管立案调查但还没 ST 的）没有被过滤。

**问题 R2 (严重) — 缺少最大回撤保护**
有日内熔断（3%），但没有总账户最大回撤保护。如果连续亏损导致总资产回撤超过 10%/15%，应该强制停止交易。

**问题 R3 (中等) — 情绪周期判断不够动态**
当前市场环境只用 MA20 判断（牛/震荡/熊），但短线交易更需要的是情绪周期判断：
- 冰点期：空仓等待
- 启动期：小仓位试错
- 发酵期：正常仓位
- 高潮期：减仓锁定利润
- 退潮期：清仓观望

复盘报告中有情绪周期判断，但没有注入到盯盘和策略管线的实时风控中。

**问题 R4 (中等) — 缺少连续亏损保护**
连续亏损 3-5 次后应该降低仓位或暂停交易，但当前没有这种机制。

**问题 R5 (轻微) — 止损止盈缺少时间维度**
持有超过 5 天还没达到止损或止盈的票，应该重新评估是否继续持有。

### 五、稳定盈利保证评估

#### ⚠️ 核心问题

作为一个交易系统，当前版本**能保证流程的完整性和执行的纪律性**，但**不能保证稳定盈利**。原因：

1. **AI 推荐质量不确定**：AI 的选股能力取决于 prompt 质量和数据质量，没有经过长期回测验证
2. **回测框架未完成**：`analysis/backtest/` 只有骨架（"骨架"注释），没有实际的回测引擎
3. **缺乏胜率/盈亏比统计**：虽然有股票追踪（tracker），但没有系统的胜率和盈亏比分析
4. **模拟盘与实盘差距**：模拟盘没有滑点、排队、涨停买不进、跌停卖不出等实际约束
5. **缺少交易心理控制**：系统没有"连亏后强制休息"、"盈利后不膨胀"等心理纪律机制

#### ✅ 已有的稳定盈利基础

1. **严格的止损纪律**：有止损/止盈/回撤止盈/时间止损
2. **仓位管理**：动态仓位控制，不盲目满仓
3. **风险过滤**：黑名单、硬关卡、严重风险过滤
4. **复盘校准**：每日复盘，AI 自我校准
5. **纪律执行**：系统自动触发信号，减少人为情绪干扰

### 核心建议

1. **优先修复严重问题**：特别是数据库索引、HTML 转义、黑名单、daily_pnl 语义
2. **拆分大文件**：watcher.py 和 analyzer.py 是首要目标
3. **补全监控盲区**：集合竞价、涨停板、盘中新闻、尾盘决策
4. **强化风控体系**：最大回撤保护、连续亏损暂停、情绪周期动态化
5. **完成回测框架**：没有回测验证的策略无法保证长期有效
